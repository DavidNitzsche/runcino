# Open question · workout_spec vs sub_label disagree on quality rows

**Filed:** backend agent · 2026-06-02
**Severity:** Medium · data integrity bug. Reader (UI) shows one
workout, watch + coach engine see a different workout.
**Owner:** backend agent (generator + prescription resolver)

---

## The bug

On David's plan (and any plan generated by `lib/plan/generate.ts`),
quality rows have a `sub_label` and `workout_spec` that describe
DIFFERENT workouts.

### Example · 2026-06-02 row

```
sub_label:       "4×1 mi @ I · 3 min jog"

workout_spec: {
  kind: "intervals",
  rep_count: 5,
  rep_distance_mi: 0.62,    ← 1km, not 1 mile
  rep_rest_s: 90,
  warmup_mi: 1.5,
  cooldown_mi: 1.0,
  rep_pace_s_per_mi: 416,
}
```

The sub_label promises 4 × 1 mile reps with 3-minute (180s) jog
recovery. The spec prescribes 5 × 1km reps with 90s rest. Different
total distance, different rep length, different recovery time.

---

## Why

Two independent generators feed the row:

1. **Prescription resolver** (`lib/plan/generate.ts:400-416`)
   builds the sub_label from a template catalog · per-distance like
   "HM intervals = 6×800m · 90s jog" or "M intervals = 5×1mi · 2:00
   jog". This goes into `sub_label`.

2. **Spec builder** (`lib/plan/spec-builder.ts:144-162`)
   autogenerates `workout_spec` from `type` + `distance_mi` + LTHR
   with hardcoded rep parameters:
   - intervals: `rep_count=5`, `rep_distance_mi=0.62`, `rep_rest_s=90`
   - threshold: `rep_count=4`, `rep_distance_mi=1.0`, `rep_rest_s=60`

The two NEVER consult each other. The sub_label is a human-readable
prescription tag · the spec is a machine-readable execution
prescription · they were grafted on independently and have drifted.

---

## Why it matters

- **Today card** reads sub_label · shows "4×1 mi @ I"
- **Watch** reads workout_spec · runs 5×1km with 90s rest
- **Run-recap composers** read both · the "did you hit the workout"
  check is ambiguous because the planned target depends on which
  side you're looking at

David's flag this morning was about distance_mi not matching the
sub_label. That fix landed (lib/plan/spec-builder.ts ·
totalDistanceMiFromSpec). But the deeper sub_label/spec drift means
even after the distance reconciles, the workout DESCRIPTION still
doesn't match the workout STRUCTURE.

---

## Options to fix

### A · Spec builder reads the prescription template (preferred)

Parse the prescription string (e.g. "5×1km @ I pace · 2:00 jog") into
its structured form and use those values in the spec instead of the
hardcoded ones. Keeps the resolver as the source of truth ·
spec-builder becomes a downstream consumer.

Pro: one source of truth (the prescription template).
Con: requires a robust mini-parser for the template strings, which
have varied shape ("6×800m", "4×1mi", "5×1km", "3×1mi", "8×400m",
etc.). Can be done with a tight regex set.

### B · Resolver builds the spec, sub_label is a derived label

Reverse the flow. The prescription resolver writes both the spec AND
the sub_label, so the sub_label is just `formatSubLabel(spec)`.
Catalog of prescriptions becomes structured (`{ reps, repMi, paceTag,
restS }`) instead of strings.

Pro: cleaner long-term. No string parsing.
Con: bigger surface · the workout_library row shape changes, the
inlinePrescriptions catalog needs to be restructured, and the catalog
already has different prescription templates per distance category.

### C · Status quo + add a generator-level invariant check

Leave the two paths · add a unit test or runtime check that catches
the mismatch at insert time. Buys time but doesn't fix the bug.

---

## Recommendation

Option A · smallest blast radius. Add a `parsePrescription(s: string)`
helper that handles the catalog's actual shapes:

```
/(\d+)\s*[×x]\s*(\d+)\s*(m|mi|k|km)/   →  { reps, repValue, unit }
/·\s*(\d+):?(\d{0,2})\s*jog/           →  { restS }
/·\s*(\d+\.?\d*)\s*min jog/            →  { restS = m × 60 }
```

Then spec-builder reads the prescription instead of hardcoding.
