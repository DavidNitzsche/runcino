# Onboarding · master spec · any user, any goal

**Status:** design brief · ready for execution-brief paring
**Pairs with:** `designs/briefs/onboarding-master-execution.md` (engineering split)
**Author:** backend agent
**For:** design agent + iPhone agent + web agent

This is the canonical spec for ALL onboarding. Any user. Any goal. Any distance from 5K through ultra. Cold-start with zero data through returning-runner with seven years of Strava history. Replaces the ad-hoc "wife is 5K/10K, here's a flow" framing with a universal coverage matrix.

It assumes the runner's first session is the most fragile moment in the relationship. Every question we ask must earn its place. Every input the engine actually needs must have a fallback that doesn't dead-end the runner. Every plan must be honest about the data it was built from.

---

## North star

> "I opened it, told it about me in 90 seconds, and it built a plan I trust."

If we make them work harder than 90 seconds before they see real value, we lose them. If we ask less than the engine needs to be honest, we ship a plan that drifts inside a week. The whole spec balances those two pressures.

Three things must be true on `/today` 10 seconds after they tap "Start training":

1. **The plan exists.** Not a placeholder — a real plan with real workouts on real dates, generated by `lib/plan/generate.ts`.
2. **The voice frames itself.** The first morning brief reads "here's what I assumed, here's what I'm watching for, tell me when I'm wrong." Not "GOOD MORNING DAVID."
3. **The path forward is honest.** If we don't know their VDOT, we say so. If we built off three Strava runs, we say so. If we have nothing and assumed everything, we say so AND offer the calibration run.

## The 5 approved design decisions

Re-stating the previously-approved direction so this brief is self-contained:

1. **Goal-time = hard + safe pace.** PR seekers get the target pace exactly. First-timers get an achievable pace (the "you'll finish") plus a stretch pace (the "if it's a great day"). The Today poster shows both bands until race day.
2. **First-race coach voice adapts.** The voice softens when the runner has zero race history and the VDOT confidence band is wide. Tightens as race history accumulates. Doctrine reference: "calibration mode" → "guided mode" → "challenge mode" tied to confidence.
3. **No race history → calibration prompt.** First easy run becomes "let's pace your first easy run together." The runner taps "start," we read pace + HR drift live, and the next run's prescription is anchored to that calibration instead of textbook defaults.
4. **Skip-all path is valid.** Connect zero sources, skip race, name + tz only → land on Today with minimum signals. Coach voice frames the absence honestly. Plan is a generic aerobic block. First check-in invites them to add what's missing.
5. **First-time-open coach copy frames it.** Morning briefing #1 starts with what we assumed and what we're watching. Not a greeting. Not motivation. A meta-line: "Today's plan is built on `[3 Strava runs / your chip answers / nothing real yet]`. I'm guessing at your easy pace — tell me when I'm wrong."

## Universal inputs catalog · criticality tiers

Every input the engine touches, classified by what BREAKS if we don't get it.

### Tier 0 · cannot proceed without

| Input | Where | Fallback |
|---|---|---|
| **Name** | Step 3 | None — explicit input. Without it the morning brief greeting can't happen. (We accept first-name only, no validation.) |
| **Timezone** | Step 3 (auto-detected via `Intl`) | UTC fallback if detection throws |
| **Goal type** | Step 1 (race distance OR "no specific race") | None — without this we can't pick a plan template |
| **Race date** (if race goal) | Step 1 | None on race path. Hard-required. |

### Tier 1 · plan honesty degrades without

| Input | Where | Fallback ladder |
|---|---|---|
| **Recent running volume** (mi/wk last 4 weeks) | Strava → Step 1b chips → manual entry post-onboarding | If absent, generator picks beginner template + writes audit row "anchored to default — no history" |
| **Longest recent run** | Strava → Step 1b chips → manual | If absent, peak long-run cap = 50% of weekly target (conservative) |
| **Years running** (experience) | Step 1b chip → profile inference from Strava volume | If absent, `experience_level=null` → coach uses neutral voice |
| **Goal time** (race goal) | Step 1 input (optional) | If absent, generator runs in "finish honestly" mode + projects a finish band |
| **Time-trial goal** (no-race path, optional) | Step 1b chip | Biases quality mix — without it the maintenance plan runs neutral |

### Tier 2 · physiology unlocks specific coaching

| Input | Where | What it unlocks |
|---|---|---|
| **Birthday → age** | Step 3 (optional) | Age-graded HR zones, age-graded VDOT projection, the 220-age fallback for `hrmax` when no observed max exists |
| **Biological sex** | Step 3 (optional) | Research/13 cycle-phase HRV adjustment, sex-specific RHR baselines, screening thresholds |
| **Height cm** | Step 3 (optional) | Cadence overstriding threshold (Research/16, /21) · cadence-under-fatigue chip |
| **LTHR** | NOT asked at onboarding | Set later via settings or auto-calibrated from a race effort. Coach defers HR-zone prescriptions until present. |
| **Max HR observed** | NOT asked at onboarding | Auto-ratcheted from watch data via `loadEffectiveMaxHr()`. Coach defers HR-zone work until 30 days of watch data lands. |
| **RHR baseline** | Apple Health → manual entry post-onboarding | Without it, readiness pillar shows "no data" + we never compute drift |
| **HRV baseline** | Apple Health · 14 nights minimum | Without it, readiness skips HRV pillar |

### Tier 3 · adapts the experience but never blocks

| Input | Where | What changes |
|---|---|---|
| **Schedule constraints** (no weekends, MWF only, etc.) | NEW · Step 1b extension | Generator's `qualityDows` + `longRunDow` + `restDow` swap |
| **Active niggle / injury** | NEW · Step 1b or Step 3 | Plan defers high-impact days · niggle banner active from day 1 |
| **Strength training habit** | NEW · Step 2 add | Strength recommender starts ON or OFF based on signal |
| **Race history** (PRs at any distance) | NEW · Step 1b add | Anchors VDOT confidence band · "first race" vs "experienced racer" voice |
| **Coaching preference** (data-heavy vs feel-first) | NEW · Step 3 add | Voice verbosity + chart depth on Today |
| **Shoe inventory** | NOT at onboarding · later | Shoe tracker shows nothing until the runner adds shoes from /shoes |

### Tier 4 · derivable, never asked

| Derived | From | Notes |
|---|---|---|
| **VDOT seed** | Goal time + race distance OR best Strava effort | Computed server-side post-onboarding |
| **`autoDetectLevel`** | `state.volume.weeklyAvg4w` + history chips | Beginner / intermediate / advanced |
| **Daily strength day picks** | `recommendStrengthDays` · runs after first plan lands | Surfaces as "+ STRENGTH" chips on week strip |
| **Cycle phase** (when sex=female) | HK push or manual entry | Optional · gates luteal-phase HRV adjustment |

## Distance coverage · 5K through ultra

Today's `RaceDistance` type:

```ts
export type RaceDistance = '5k' | '10k' | 'half' | 'marathon' | 'none';
```

**Gap:** ultras (50K / 50mi / 100K / 100mi) are not representable. The brief proposes:

```ts
export type RaceDistance =
  | '5k' | '10k' | 'half' | 'marathon'
  | 'ultra-50k' | 'ultra-50mi' | 'ultra-100k' | 'ultra-100mi'
  | 'none';
```

### Per-distance plan generator behavior

| Distance | Plan template | Quality mix | Long-run cap | Block length |
|---|---|---|---|---|
| 5K | VO2max-leaning · sharp tempo + intervals | 60% VO2max / 30% threshold / 10% race-pace | 6-10 mi | 8-12 weeks |
| 10K | Threshold + VO2max split | 40% VO2max / 40% threshold / 20% race-pace | 8-12 mi | 10-14 weeks |
| Half | Threshold-heavy · marathon-pace long runs | 20% VO2max / 50% threshold / 30% MP | 12-16 mi | 12-16 weeks |
| Marathon | Aerobic + threshold + MP long runs | 10% VO2max / 35% threshold / 55% MP | 18-22 mi | 16-20 weeks |
| Ultra-50K | Aerobic base + back-to-back longs + nutrition emphasis | 5% VO2max / 25% threshold / 70% Z2 | 22-26 mi · b2b long pair | 16-22 weeks |
| Ultra-50mi | Same as 50K + multi-hour Z2 emphasis + power-hiking | 5% / 20% / 75% | 26-30 mi solo · 18+18 b2b | 18-24 weeks |
| Ultra-100K | Multi-day blocks · back-to-back-to-back possible | 0% / 15% / 85% | 30+ mi solo · stacked weekends | 20-26 weeks |
| Ultra-100mi | Nutrition/hydration coaching takes over · sleep deprivation prep | 0% / 10% / 90% | 30 mi solo · sequential weekends | 24-30 weeks |
| No race | Maintenance · 1 quality/week · 1 long/week | 100% aerobic w/ 1 lever | 50% of weekly target | 16 weeks (rolling) |

**Honest truth:** the doctrine for ultras is thinner in Research/. The brief authorizes a `template_v1` for 50K/50mi using Magness + Koop, and explicitly defers 100K/100mi to a v2 with stricter doctrine review. For v1, picking ≥100K writes a coach intent "experimental template · doctrine reference Koop §Multi-day blocks" and surfaces a warning chip on /today.

## Path map · who goes where

```
                            ┌── Strava connected · 100+ runs ─→ "Returning runner" path
LandingHero ──→ Step1Goal ──┤
                            ├── Strava connected · <100 runs ──→ "Light history" path
                            │
                            ├── HK connected · no Strava ──────→ "HK-only" path
                            │
                            └── Nothing connected ─────────────→ "Cold-start" path
                                       │
                                       ├── Race goal ───→ Step1 → Step2 → Step3 → Plan
                                       │
                                       └── No race ─────→ Step1 → Step1b → Step2 → Step3 → Plan
                                                                          │
                                                                          └── Skip-all → Plan + voice frame
```

### Returning runner (Strava 100+ runs)

- Step 1b history section pre-fills from Strava silently · no chip group rendered unless they tap "edit"
- VDOT confidence is HIGH · coach voice is direct
- First plan generates with the runner's actual peak long-run as `peakLongRunMi` floor
- Skip-all from this path is hidden · they connected Strava, the plan has real data

### Light history (Strava <100 runs or only a few months)

- Step 1b history section pre-fills, but with a note: "We have 3 weeks of Strava — extending the baseline takes a few more runs"
- VDOT confidence is MEDIUM · coach voice hedges paces ("aim for 8:30 · I'll tighten this after a couple more easy runs")
- First plan generates with safety margins · prescribed paces 5-10s slower than VDOT-derived

### HK-only (no Strava)

- Step 1b history section asks for chips (no Strava to pull from)
- VDOT seed comes from goal time (race path) or TT goal (no-race path) — never from HK
- HK contributes sleep / HRV / RHR baselines from day 1 (if 14+ nights logged)
- First plan generates from chip-reported volume · marked "anchored to your answers" in audit row

### Cold-start (no connections, no history)

- All chips required in Step 1b
- VDOT seed comes from goal time OR a default per distance + age + sex
- **Calibration prompt fires on Today day 1**: "Let's pace your first easy run together"
- First plan generates with the most conservative template per distance
- Coach voice is in "calibration mode" for first 2 weeks · all paces have a "+ or - 15s" band
- Skip-all valid → land on Today with minimum signals + calibration prompt

## Race history capture · NEW

Currently no race history is captured at onboarding. The 5 approved decisions all reference race history depth, so this brief adds it.

### Step 1b extension · "Have you raced before?"

After the time-trial section, before the weekly target section:

```
Race history · optional

  [ No, this would be my first ]
  [ Yes — what's your PR? ]

When "Yes" tapped:
  Distance chips: 5K · 10K · Half · Marathon · Other
  Time input (chip ladder per distance, free text on Other)
  When raced chips: < 6 months / 6-12 months / 1-2 years / 2+ years
  + Add another  (up to 3 entries)
```

### What the engine does with race history

| Race history | VDOT confidence | Coach voice | First plan |
|---|---|---|---|
| First-ever race | LOW · wide band | calibration mode | most conservative template |
| 1 race · 1+ year ago | LOW · wide band | calibration mode w/ historical anchor | conservative · references the old PR as floor |
| 1 race · recent | MEDIUM | guided mode | template-anchored to PR-derived VDOT |
| 2+ races · recent | HIGH | challenge mode | template at goal-time-derived VDOT |

### Storage

- `race_history` array on `profile` · `[{distance, timeSec, raceDate, source: 'self_reported'}]`
- VDOT seeder reads `race_history` first, falls back to goal time, falls back to default-per-distance
- Coach voice band exposed as `state.voiceBand: 'calibration' | 'guided' | 'challenge'` (new field on CoachState)

## First-week coach voice adaptation

The voice band drives copy across THREE surfaces:

1. **Morning brief headline** (`brief.headline` on Today's readiness card)
2. **Run-day pre-run cue** (`brief.preRunCue` on the workout poster)
3. **Post-run recap framing** (`runRecap.framing`)

### Band definitions

| Band | Trigger | Tone | Example morning headline |
|---|---|---|---|
| **calibration** | First-ever race OR no race history OR `vdot.confidence < 0.4` | "Let's figure this out together. I'll adjust as I learn." | "Day 1 · let's see what easy feels like for you." |
| **guided** | 1 recent race OR `vdot.confidence 0.4-0.7` | "Here's the plan. Tell me if a pace feels wrong." | "Easy 4 today · aim for around 8:45, ±15s is fine." |
| **challenge** | 2+ recent races OR `vdot.confidence > 0.7` | "Hit the prescription. The plan is honest." | "Easy 4 at 8:45 · banking aerobic for Thursday." |

### Band transitions

The voice band recomputes daily. Triggers:

- New race result lands → band steps up (typically guided → challenge)
- Goal time wildly off projected (>10%) for 2+ weeks → band steps down (challenge → guided)
- Calibration run pace + HR captured → band can step calibration → guided immediately
- Subjective check-in disagrees with objective for 5+ days → coach voice softens regardless of band

### "I assumed X, watching for Y" meta-line

The first morning brief (Day 1 only) renders an extra line above the headline:

```
Based on [your 18 Strava runs / your chip answers / first-run defaults], I'm
assuming your easy pace is around [8:45]. I'll tighten this after your first
easy run — tell me if it feels wrong.
```

The bracket variables come from:
- "your N Strava runs" if `recentRuns.length > 5`
- "your chip answers" if `histAvg` is set but no Strava
- "first-run defaults" if neither

## Skip-all path · valid landing

When the runner skips every connection AND skips race AND just hits Continue through Step 1b with chip answers, the system MUST still ship a usable plan and a usable Today.

### What we ask, minimum

- Name (Tier 0)
- Timezone (Tier 0)
- Goal type (Tier 0) — `none` is fine
- Step 1b chips: weekly mileage + frequency (Tier 1) — to anchor the maintenance template

### What we DON'T need

- Strava (skipped) — plan generates from chip answers
- HK (skipped) — readiness shows "no data" pillars cleanly
- Birthday / sex / height (Tier 2) — coach defers HR-zone prescriptions
- Race history (Tier 3) — voice runs in calibration mode

### Today day 1 · skip-all view

The Today screen renders normally with three new pieces:

1. **Calibration banner** at the top: "Let's pace your first easy run together. Tap to start."
2. **Empty readiness card** with the connect prompt: "Connect Apple Health to start your baseline."
3. **First workout in calibration framing**: "Easy 3 · with you. We'll learn your pace as you go."

### Calibration prompt mechanics

When the runner taps "start" on the calibration run:

- A `calibration_session` row is created (new table)
- Watch / Strava write the run normally
- On run completion, the engine reads pace + HR drift across miles 1-3
- Computes calibrated `easyPaceTarget` ± confidence band
- Writes a coach intent `calibration_completed` with the derived band
- Next morning brief uses the calibrated band, voice transitions to "guided"

If the runner skips the calibration run (just runs without tapping start) — the engine still uses pace + HR from that first run for calibration, with a slightly wider band.

## Edge case matrix

| Scenario | What we do |
|---|---|
| Runner picks race < 2 weeks out | `generatePlan` returns `ok: false` with reason · race row saved, plan deferred · Today renders "race too close · here's how to taper from here" |
| Runner picks race > 1 year out | Same · race saved, plan deferred · Today shows "we'll build the plan when we're within range" + maintenance block in the meantime |
| Runner picks two races (entered "Add another A-race") | Only A-race priority counted at onboarding · second race added later via /races · `horizonRaise` rule handles the long-run cap (existing) |
| Runner enters goal time slower than current Strava VDOT projects | Plan generates to the goal · coach intent `goal_slower_than_projected` fires Day 1 with "let me know if you'd like to bump this up" |
| Runner enters goal time wildly fast (>10% below VDOT) | Plan generates with caps · audit row `goal_aggressive · vdot_says_X · stretch_band_added` · stretch pace shows on poster |
| Runner reports niggle on Step 1b | Plan defers high-impact days · niggle banner active from Day 1 · "we'll check in on the [body part] each morning" |
| Runner reports schedule constraints (can't run weekends) | `qualityDows + longRunDow + restDow` swap · long-run lands on Friday/Monday per pref · audit row records the constraint |
| Runner is sub-13yo or over 100 (birthday math) | Age input ignored · profile.age = null · coach voice never references age |
| Runner enters birthday in future | Ignored silently · same as above |
| Runner is on web only, no iPhone | HK + Watch tiles say "Add later from iPhone" instead of "Open on iPhone" · plan ships without them |
| Runner is on iPhone, no Strava | Lilian deck still works · history pulls from HK workouts (running session type) if connected |
| Runner ALREADY had an account (returning user via Strava sign-in) | Skip onboarding · land on /today · existing data populates everything · NEW: detect missing fields on first morning brief and surface as "we don't have your birthday — tap to add" |
| Runner abandons mid-flow (closes tab on Step 2) | URL has their answers · returning to /onboarding picks up where they left off · once Step 3 completes the txn fires |
| Runner picks ultra distance | v1 template generates · warning chip on /today "experimental template" · existing doctrine citations point at Magness/Koop |
| Runner picks "no specific race" but enters a TT goal | Plan biases quality toward TT distance · maintenance plan tagged "TT-leaning" in audit |
| Runner is female + opts in to cycle tracking | Cycle phase tile renders on Health page · luteal-phase HRV adjustment active · onboarding doesn't ASK about cycle (privacy) — opt-in lives in Settings |
| Strava OAuth fails mid-onboarding | Step 2 surfaces "Couldn't connect Strava · try again or skip" · continue button stays active · runner can complete without |
| Runner taps skip-all then comes back next day and adds Strava | Plan regenerates via lifecycle · `replan_event=strava_connected` · audit row + coach intent surface the change |
