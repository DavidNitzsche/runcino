FAFF · WATCH FACE · 2026-05-28

WHEN THE WATCH SPEAKS.

Wrist-level surface for the runner mid-effort. Same Oswald display recipe, same seven semantic colors, same dark canvas. Five faces — one per workout type — each glanceable inside half a second at race pace. The watch app stays preserved per Cardinal Rule #2 · this deck is the face contract the existing watchOS app renders against.

RULE 1 · BIGGEST NUMBER
Always Oswald 700 · tracking -0.015em · line-height 0.86 · tabular-nums. The hero stat dominates. Nothing else competes.
RULE 2 · UNDER 0.5s
Glanceable while running. No scroll. No swipe to read. Three zones: TOP context, MID hero stat, BOTTOM secondary.
RULE 3 · STATE-CODED
Background tint comes from the day-state gradient at 10-18% opacity. Hero color is semantic: green in-band, amber drifting, red over.
RULE 4 · TOP-BOTTOM SYMMETRY
Honors the locked Runcino NumberFace law — top margin equals bottom margin. Small caps top-label rides the OS clock baseline.
RULE 5 · NO EMOJI · NO LLM
Every label is deterministic. Every threshold cites doctrine. No on-watch coaching prose — just numbers + state.
RULE 6 · PRESENT · DO NOT COLLECT
The watch already gathers HR + pace + distance + route via HKWorkoutSession. The face's only job is to render what's already in CoachState.

WHAT EACH ZONE DOES

In-band · respect the cap.

Easy days are won by not racing them. The runner's pace band is set by the plan engine (VDOT-driven per Daniels). Hero color is the only thing the eye needs to lock onto — green means you're inside the band, no work needed.

TOP ZONE · static
EASY label + distance remaining. Never changes during the run except the live mile counter.
MID ZONE · live
Current pace. Green in-band · amber ±10s drift · red over the band ceiling.
BOT ZONE · live
Current HR + cap reminder. Cap text flips red when HR exceeds the LTHR-derived ceiling.
CANVAS · subtle
--g-easy at 10% opacity. Just enough to identify the day-state without competing with the hero.

REPS · TARGETS · REST

Three contexts · one face.

A 6×800m session has three modes the face must handle: running a rep, recovering between reps, and the rep target reminder before it starts. The static top label carries rep number + target; the bottom row swaps live between rest-countdown and an empty (dim) state.

TOP ZONE · static
REP n/m + the rep's target. Set by the plan engine. Doesn't change mid-rep.
MID ZONE · live
Rep elapsed time (big) + current pace per mile (sub). Pace stays amber in target band.
BOT ZONE · contextual
REST n:nn countdown between reps. Fades to dim once the next rep starts.
CANVAS · amber
--g-quality at 14% — slightly warmer than Easy because the work-rest rhythm signals "high-intent" day.

AVERAGE · FUEL · HEAT

Hours · not minutes.

A long run is a thermal + glycogen story. The face surfaces what matters at that timescale — rolling average pace, the next fuel cue, and a heat advisory if HR is drifting up against pace (a classic dehydration / heat-strain signal).

TOP ZONE · static
Distance progress + HR readout with cap. Two numbers up top only on Long — the runner has time to read them.
MID ZONE · live
Rolling average pace (not instantaneous). Color stays blue across the band — it's a steady-state effort.
FUEL SUB · plan-driven
Fuel cue from plan. Comes from the runner's race nutrition profile if set, defaults to every ~45 min otherwise.
BOT ZONE · contextual
Heat chip fires when HR drift > 8bpm vs same-pace baseline. Hidden when not relevant.

SPLIT · PROJECTION · DELTA

One mile at a time.

On race day the runner needs three things: how am I doing right now, how is the projection tracking, and am I ahead or behind goal. Hero is mile split (the rhythm the race is run in), color-coded green if under target, amber if at target, red if over.

TOP ZONE · live
Current mile marker + projected finish computed off rolling average. Updates every 30s, debounced.
MID ZONE · live
Mile split for the just-completed mile. Green under target · amber at · red over.
BOT ZONE · live
Split delta vs the race plan. Negative banked time = green. Positive lost time = red.
CANVAS · race-orange

SECTION · ARCHITECTURE

PRESENT · NOT COLLECT.

Per Cardinal Rule #2, the existing watchOS app is preserved. The watch already owns the HKWorkoutSession lifecycle, the route polling, the HR stream, and the live-distance accumulator. The face's only job is to render. Five architecture decisions land below.

WATCH ALREADY DOES

Collection · system layer.

  • ·HKWorkoutSession lifecycle — start, pause, resume, end, file write. Untouched.
  • ·Heart rate stream via HealthKit at 1Hz. Cached in CoachState.hr.
  • ·Pace + distance from CMPedometer + GPS fused per the existing logic. Lives in CoachState.pace and CoachState.distance.
  • ·Route polling via CLLocationManager for the post-run map. The face never renders the route.
  • ·WatchConnectivity sync from iPhone — plan + targets + LTHR cap arrive before workout start.
FACE'S ONLY JOB

Render · view layer.

  • ·Select the face by workout type (easy / quality / long / race / done) sent from iPhone.
  • ·Read CoachState + the active plan target. Pure read · no mutation.
  • ·Compute color encoding — in-band / drift / over — from CoachState and the workout target.
  • ·Render three zones per the face spec — top, mid, bot. Static labels stay static · live numbers update every 1s.
  • ·Hand off pause / end back to the existing HKWorkoutSession lifecycle. No new state.

FOOTER · OPEN DESIGN QUESTIONS

FIVE THINGS UNSETTLED.

The face shape is locked. These are the design decisions still on the table — most are about the edges (rare states, runner preference, device variants). None block v1 implementation against the existing watch app.

Q1
Race-face split delta direction. Should negative time vs target read as "−0:04 banked" (math-true) or "+0:04 ahead" (intuitive)? Currently shown as math-true negative → green · gut says runners read "+" as "ahead." Needs a test.
Q2
Heat-chip threshold authority. The Long face chip currently fires on HR drift >8bpm vs same-pace baseline. Should the threshold flex by ambient temperature input from HealthKit, or stay deterministic? Cardinal Rule #3 prefers cited doctrine.
Q3
Quality-face rep-target visibility. Mid-rep, does the runner want the next rep's target visible (predictive prep) or stay focused on the current rep elapsed? Tested both on the Runcino watch face contact sheet — no clear winner.
Q4
Aluminum vs Ultra layout. The face is designed for 49mm Ultra (198×242 css). Aluminum 41mm and 45mm are 162×197 and 184×224. Auto-shrink the hero number or auto-drop the bottom row? The Runcino law says width-cap by widest row — needs porting.
Q5
Recovery face auto-dismiss. How long does Face E stay visible post-finish? 90 seconds default but the runner may still be lying on the ground for minutes. Tap-to-dismiss vs timeout vs both? Currently spec'd as both (timeout 90s · tap anytime).
Q6
Complication availability matrix. The two glances are designed for the modular corner slot. Should they also render at extraLarge, circular, and graphicBezel? watchOS makes us pick — minimum viable is corner only, full coverage is more design work.
