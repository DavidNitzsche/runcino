FAFF · NOTIFICATION SYSTEM · DESIGN DECK · 2026-05-28

When Faff
speaks up.

Push notifications are coaching beats, not noise. Each one earns its tap by answering a real question the runner already has: is it race day? · am I still skipping? · did the week land? · is the niggle better? Seven categories. Deterministic templates. Zero LLM in the payload. Quiet hours respected. Race-day delivery is non-negotiable.

LLM IN CONTENT
Zero. Ever.

Short.
Direct.
Plain.

Faff's voice on the lock screen is the same as in the app — verb · noun · numbers. Lowercase prose. SCREAMING CAPS only on the title line, and only when the data warrants it. No "Don't miss your run!" No "Crush it today, runner!" The runner already knows.

1.Deterministic templates only. Slots are filled from typed state — no generative prose.
2.One sentence. Maybe two. If it needs three, it doesn't belong on the lock screen.
3.Numbers are tabular-nums. 19.7 / 43.8 mi, not about 20 miles.
4.No urgency without cause. No clock emoji. No exclamation points unless the runner finished a PR.
5.Race-day is the one place we go slightly louder. Everywhere else is calm.

ARCHITECTURE · BEHIND THE PUSH

Token in.
Beat out.

1 · TOKEN REGISTRATION

App boot → token → server.

When the iOS app launches and the user has granted notification permission, we register for remote notifications and POST the resulting device token to our backend. Re-runs on app foreground in case Apple rotates.

# on app boot, after auth UIApplication.registerForRemoteNotifications() ↓ didRegisterForRemoteNotifications(deviceToken) ↓ POST /api/notifications/register { user_id, device_token, platform: "ios", app_version, bundle_id } ↓ device_tokens table · upsert by token { token, user_id, last_seen_at, revoked_at: null }
2 · SCHEDULER · HYBRID

Cron + queue.

Two engines, one job. Cron fires the time-based categories (B / D). A queue consumes events from day_states, health_events, races, and strava_sync_failures for the rest. Both feed the same delivery worker.

cron (every 5 min) ↓ scans users w/ pref + tz + race calendar ↓ enqueues: A · B · D · F (race-countdown) event bus ↓ day_state changes → C ↓ health_event open → E ↓ streak threshold → F (streak) ↓ strava 401 ×3 → G delivery_worker (single point) ↓ dedup check · quiet hours filter ↓ render template (NO LLM) ↓ APNs HTTP/2 push · token from registration
3 · PAYLOAD SHAPE

aps + faff.

Standard APNs JSON. The aps dict gets the OS-visible content. A sibling faff dict carries metadata for in-app routing, deduplication acks, and category-specific slots.

{ "aps": { "alert": { "title": "RACE DAY · AMERICA'S FINEST CITY", "body": "Gun 7:00. Uber pickup 6:25..." }, "sound": "default", "category": "FAFF_RACE_DAY", "thread-id": "race-{race_id}", "interruption-level": "time-sensitive" }, "faff": { "kind": "race-day", "race_id": "afc-2026", "deeplink": "faff://races/afc-2026", "dedup_key": "race-day:afc-2026", "fired_at": "2026-06-07T05:30:00-07:00" } }
4 · RICH NOTIFICATION CATEGORIES

Inline actions registered at launch.

iOS requires that interactive notification actions be registered at app launch via UNUserNotificationCenter.setNotificationCategories. Each category maps to one of our seven kinds. Action handlers POST to small ack endpoints.

FAFF_RACE_DAY · 1 action · OPEN FAFF_RACE_EVE · 1 action · CHECKLIST FAFF_SKIP_RECOV · 2 actions · READY · STILL FAFF_WEEKLY · 3 actions · SOLID · TIRED · WRECK FAFF_NIGGLE · 4 actions · BETTER · SAME · WORSE · GONE FAFF_MILESTONE · 0 actions · soft beat FAFF_STRAVA_RECON · 1 action · RECONNECT # ack endpoint POST /api/notifications/ack { dedup_key, action, fired_at }
5 · DEDUP + QUIET HOURS

Filter before send.

Every payload runs the dedup gate before APNs. Same key in the prior 24h → drop. Quiet hours: scheduled time gets shifted to next allowed minute. Race-day morning bypasses both because the runner has explicitly opted in.

render_template(kind, slots) ↓ dedup_gate(dedup_key) ↓ if seen in 24h → DROP, log silently ↓ quiet_hours_gate(user_tz, now) ↓ if inside [22:00, 06:00) ↓ and kind != RACE_DAY ↓ → defer to next 06:00 ↓ APNs HTTP/2 push (token, payload) ↓ notifications_log table { user_id, kind, dedup_key, fired_at, apns_response, ack_at, ack_action }
6 · APPLE DEVELOPER SETUP

Cert in the right place.

APNs auth uses a token-based key (.p8) generated from the user's Apple Developer account. Stored as a backend secret. We do not use legacy certificate-based auth — token-based is what Apple recommends and works for both sandbox + production.

# one-time setup 1. developer.apple.com → Keys → New 2. Enable APNs · download .p8 · note Key ID 3. Note Team ID from membership page 4. Store as env: APNS_KEY_ID=ABC123XYZ APNS_TEAM_ID=TEAM456789 APNS_KEY_P8=----BEGIN PRIVATE KEY----... APNS_BUNDLE_ID=run.faff.ios # sign each JWT per request JWT header: { alg: ES256, kid: $APNS_KEY_ID } JWT claims: { iss: $APNS_TEAM_ID, iat: now } Authorization: bearer <signed jwt>

OPEN DESIGN QUESTIONS

Still
undecided.

1

Does the race-day notification fire on Apple Watch even if the phone is in Do Not Disturb?

APNs lets us flag interruption-level: critical, which bypasses every system filter — but requires an Apple-granted entitlement that's not trivial to get. time-sensitive handles most cases but a runner who set Focus to silence everything might miss the call. Walk-the-line: do we apply for critical, accept the review burden, or trust the runner's Focus settings?

2

Should Category C (skip recovery) inline-actions trigger a plan rewrite, or just log a flag?

If the runner taps STILL SKIPPING from the lock screen, the day_state is already missed. Do we re-rank tomorrow's plan immediately (potentially shifting a quality day later) or just record the signal and let the next daily plan resolver pick it up? Bias toward the latter, but it means the runner won't see plan change confirmation in the notification ack.

3

Quiet hours default — runner local vs. device local?

A runner traveling for a destination race lands in a new timezone but the device timezone often updates 20+ minutes later. Do we honor the device timezone (simpler, OS-truthful) or store the runner's home timezone and respect that (better for travel races but adds a config step)? Lean toward device, but race-day notification needs to use race city tz.

4

Should Category F (streak / milestone) be opt-in or opt-out by default?

Streaks lean Strava-coded — gamification that some runners actively dislike. Plan-following audience may find them noise. Default OFF feels right but means many runners never discover them. Compromise: surface during onboarding ("Want a beat at 7 / 14 / 30 days?") and let them choose then.

5

If APNs fails delivery (token revoked, network blip), do we retry — and how visibly?

APNs returns a per-push response. For race-day, a failed primary push triggers a +5min retry. But for the other six categories, retry cost is silent inconvenience. Do we log + drop, retry once silently, or surface "Faff couldn't reach you this morning" inside the app when next opened? Affects the trust contract.

6

Should the deep-link target depend on the day-state, or always be /today?

Category A clearly opens the race timeline. Category G clearly opens reconnect. But C, D, E all start from "open the app to see context" — should those land on /today (the runner's habitat) or the specific surface that prompted the notification (recovery card / weekly summary / niggle detail)? Argument for /today: one mental model. Argument for surface: zero extra taps.

7

Does the niggle-check notification carry the planned workout, or stay purely a body question?

Right now the mockup asks "better today?" without showing what's on the plate. Argument for adding the workout: same decision, fewer taps. Argument against: notifications stay single-purpose · runners conflate "should I run?" with "is the niggle better?" when they're separate questions.
