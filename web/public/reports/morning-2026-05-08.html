Was the easy ratio reading 100%

Fixed · Commit 831d469

Easy ratio classifier rewritten with research-anchored cascade

Old logic: hard if name matched 15 patterns OR avgHr ≥ 152 BPM. A runner doing tempo work at 145 BPM with a generic "Morning run" name got classified as easy. Activities without HR data defaulted to easy. Result: 100% easy on most real Strava feeds.

New logic, cited to Daniels' 80/20 distribution (Research/01 + intensity.ts):

  1. Name pattern — runner's intent. Regex expanded ~15 → ~30 patterns: tempo / threshold / intervals / fartlek / progression / VO2 / cutdown / ladder / track / hills / surges / pickups / cruise / sub-threshold / MP miles / marathon pace / wave tempo / 400s / 800s / 1k / 1200s / 1600s / strides / pyramid. New EASY_NAME_RE catches recovery / shakeout / base / MAF / Z2 / aerobic / chill / jog.
  2. VDOT pace zones — when current VDOT exists, M-pace-or-faster = hard, E-zone or slower = easy. Daniels' 80/20 boundary.
  3. HR threshold — fallback when no name + no VDOT.
  4. Long run default — runs ≥12 mi without quality signal → easy.
  5. Unknown bucket — explicit when no signal applies. Don't lie that it was easy.

UI now shows easy / hard / unknown as a 3-segment bar with a LOW CONF chip when classification confidence drops below 70%. "On target" verdict is phase-aware — pulls from coach-principles (TAPER 78% / PEAK 75% / BUILD 70% / BASE 80% / POST_RACE 90% / REBUILD 85%) instead of a static 75%.

Top priority for next session

Per the audit's #1 finding

HIGHEST LEVERAGE

Wire runner-profile.ts to server-side CoachState

The runner's age, sex, HRmax, and RHR live in localStorage. The dashboard reads them client-side — VDOT tile shows age-graded VDOT, HR zones tile shows BPM ranges, profile editor saves them. But the server-side coach engine never sees them.

coach.briefRaceMorning, coach.briefDailyTraining, and coach.assessReadiness all run with no profile. The brief never says "exceptional VDOT for a 60yo woman" because the route doesn't have age + sex. The engine's "yesterday hard" gate is still hardcoded 152 BPM even when the runner has set a measured HRmax of 175.

What closing this would unlock:

  • LLM brief contextualization on age + sex (prompt sees them, voice can use them)
  • HRmax-aware "yesterday hard" detection (no more hardcoded 152)
  • Karvonen / HRR zones (need RHR server-side)
  • Future age-specific recovery rules from Research/14

Two paths: (1) pass the profile from client to API on each call (header or body) — quick, no DB migration; (2) migrate to a Postgres runner_profiles table once auth is in. Path 1 is the right move for the next session; path 2 is the migration when auth lands.
