Section 01

Executive
summary

What it is. A multi-surface running app for one user (the developer). Three surfaces, one source of truth: a web command center for planning and analysis, an iPhone daily companion, an Apple Watch execution layer. An AI coach interprets the running knowledge base against the user's actual training data to produce prescriptions, recap analysis, insights, and chat answers.

The brand. Hero numbers. Small-caps gray labels. Color as semantic signal (green recovery / blue training / amber upcoming / red shortfalls / purple milestones). Coach voice over neutrality — direct, honest, specific. "Recovery is the workout. Volume drop is intentional." Density without clutter, scaled by device.

The non-negotiable loop. Plan flows web → backend → phone → watch. Activity flows watch → phone → backend → web → external services. Auto-logging is the default. The user opens the app to review a run, never to log one.

Section 02

Architecture
decisions

Four cross-cutting decisions that everything else hangs from. Each is opinionated and grounded in the deep research docs.

Recovery score

0–100 composite, four bands, three contributing factors

A composite daily readiness number with a four-band traffic light, three contributing-factor cards, and a coach voice read.

Composite = (
  0.45 × HRV_score        // 7-day rolling LnRMSSD vs. 60-day mean ± SD
  + 0.20 × RHR_score      // 7-day rolling nocturnal RHR vs. 60-day mean ± SD
  + 0.25 × Sleep_score    // (TST / personalized_need) × efficiency vs. 14-day
  + 0.10 × Temp_score     // 3-night rolling deviation, flag only beyond ±0.5°C
) × Load_modifier         // [0.85, 1.10], driven by ACWR and ATL trend

// No subjective modifier — recovery score is purely objective
// (per Decision 03). Coach interprets in narrative; user does not tap.

Bands: Red 0–33 / Yellow 34–49 / Green 50–79 / Optimal 80–100. Single source of truth across all surfaces.

Baselines: 60 days for HRV/RHR; 14 days for sleep; 28 days for training load. Score is hidden for the first 7 days of data with a "calibrating" state. When score and feeling disagree (>30-point delta), surface the disagreement explicitly. Do not average it away.

Source: D1.

Coach LLM

Anthropic Claude + hybrid retrieval + citations + voice validators

Model: Anthropic Claude Sonnet 4.6 for chat + analysis + insights. Claude Haiku 4.5 for intent detection, summarization, output validators. Anthropic prompt caching (90% input discount) makes pinning the system prompt + KB economically viable.

Retrieval: Hybrid BM25 + Voyage-3-large dense embeddings + Cohere rerank-v3 + RRF fusion. KB stored in pgvector. Intent-gated — not every message triggers retrieval.

Voice: System prompt with embedded few-shot examples. Output validator (Haiku-backed) flags responses that drift sycophantic or vague. Format: WHY / FOCUS / BACK OFF IF / YESTERDAY for inline blocks.

Safety: Five layers — (1) retrieval-grounded for facts, (2) "not a doctor" hedge on prescriptive medical, (3) refusal rules for diagnosis, (4) KB citations, (5) output validator for unsupported numerical claims.

Cost: ~$4/active-user/month at ~6 turns/day with cache hits. Heavy users $8–10.

Source: D4.

Watch active workout

Single-screen mid-run, audio + haptic over visual

One screen. Pace huge. Target delta colored. Interval banner (top), mini stats (edge), next-interval preview (bottom).

┌─────────────────────────┐
│ MILE 3 OF 5 · THRESHOLD │  interval banner
│      6:42               │  current pace (max size)
│      /mi                │
│   ▼ 7 sec               │  target delta (green/yellow/red)
│   3.21mi  21:34  168bpm │  distance / time / HR (small)
│ NEXT: 90s recovery      │  next interval preview
└─────────────────────────┘

Audio: Coach voice TTS at interval transitions, 30-sec warnings, mile splits with pace, halfway in long intervals, finish.

Haptic: Single tap = split/lap. Double = interval transition. Triple = halfway. Long buzz = off-target ≥30s. Quadruple = workout complete.

WorkoutKit: Native structured-workout support (watchOS 9+). Custom rendering on top for branded labels.

Battery: Standard GPS + AOD off by default. Below 30% auto-suggest disabling AOD, dropping to balanced GPS.

Removed · Per Decision 12

Plan Builder

Removed from scope. The coach is the planner. There is no "Build a Plan" wizard.

Onboarding asks once for goals, current fitness, and days/week available. From day one, the coach generates the next 1–2 weeks adaptively and re-plans continuously. Adding a race on the Races calendar (Surface 8) triggers race mode automatically — the coach sees the date and phases in marathon-specific work, scheduling the long-run progression and the taper backward from race day. No mode toggle. No wizard.

Where to look instead:

  • Surface 2 (Training) — the canonical view of what the coach has generated. Shows current cycle arc, next ~2–3 weeks of generated workouts.
  • Surface 8 (Races) — adding a race here triggers the coach to begin building toward it.
  • Surface 15 (Coach) — ask the coach why a workout was scheduled the way it was.

The Plan Builder wizard mockup later in this deck is obsolete and should be ignored.

Section 08 · Decisions made

Decisions

Eleven design decisions made. These are no longer open. Some simplify scope (drop Garmin/Coros, drop manual subjective check-in, drop race photos, drop the "Send to Watch" CTA). Others sharpen bets (Amp is the strength surface, the coach is fully adaptive, recovery score is purely objective). Cascading implications for earlier sections noted inline.

01 ✓
Garmin / Coros API access — DROPPED. User does not own Garmin or Coros. Apple HealthKit is the sole biometric pipeline. Removes Garmin/Coros integration scope from Phase 7 (~30% Phase 7 complexity reduction). Sync architecture (Section 02) implicitly simplifies: HealthKit + Strava + Whoop + Oura only. Plan-push-to-Garmin-device feature removed.
02 ✓
Coach voice — Direct + personality + the "why". Default position is direct; the coach is opinionated by design. Personality dial removed (no Direct↔Encouraging toggle). Every prescription includes its reasoning. Recovery is the workout. Volume drop is intentional.
03 ✓
Subjective wellness — DATA-DRIVEN, no manual input. No daily 4-tap check-in. Recovery score uses only objective signals (HRV, RHR, sleep, training-load). The coach's narrative interprets — but the user doesn't tap energy/soreness/mood each morning. Recovery algorithm (Section 02) updated: subjective modifier removed; weights re-balance to HRV 45% / RHR 20% / Sleep 25% / Training-load modifier × 1.0 / Body-temp 10%.
04 ✓
Deployment — TestFlight. Personal use only. No App Store publish. Removes subscription/billing complexity entirely. Onboarding can assume single-user.
05 ✓
LLM — Anthropic Claude. Sonnet 4.6 + Haiku 4.5 stack confirmed. Voyage-3-large embeddings, pgvector, Cohere rerank-v3.
06 ✓
Race photo integration — SKIPPED. Drop MarathonFoto / Sportograf / FinisherPix entirely. User can manually attach photos to race detail. Phase 9 line item removed.
07 ✓
Strength equipment — AMP IS PRIMARY. User owns the Amp. Strength view (Surface 7) defaults to Amp programming with its mode vocabulary (Fixed / Band / Eccentric / Tempo / Drop Set), 5–100 lb electromagnetic resistance range, motorized cable. Cable-trainer programming is the canonical strength layer. Bodyweight-only is the travel fallback. Barbell / dumbbell programming is secondary or absent.
08 ✓
Plan generation — ADAPTIVE, not template-pick. No fixed Pfitzinger / Hansons / Daniels selection. The coach generates plans dynamically and re-plans continuously based on recovery state, completed runs, upcoming races, and the user's history. Plan Builder (Surface 17) becomes "Plan Generator" — fewer wizard inputs (goal, current fitness, days/week available), no plan-template picker. The KB methods inform the coach's reasoning (Pfitz-style for marathon-specific work, Norwegian-singles for threshold-heavy phases) but the user never picks one. Plans are versioned; every modification is a new plan version with reasoning logged. The plan you see today may not be the plan you see tomorrow — and the coach explains what changed and why.
09 ✓
Strava description default — ON. Coach analysis appears in completed-workout Strava description by default. Per-activity opt-out toggle in run detail.
10 ✓
Health data — ALL ON. Body composition, bloodwork, cycle tracking, all health categories enabled by default. No per-category opt-in friction. Privacy controls remain in Settings for any future scope where data leaves the user's account (LLM provider, Strava description).
11 ✓
"Send to Watch" — REMOVED. Auto-sync is the contract. No manual button to push today's workout to the watch. The watch auto-fetches today's planned workout in the background via WatchConnectivity + HealthKit Workout Plans. The Workout Detail screens (web Surface 3, iOS Surface 23) show a quiet status indicator instead of a CTA — "Watch ready · synced 4m ago" when fresh, "Tap to retry" only if sync is stale (>30 min) or failed. Manual push is fallback, never default. Mockups currently show "Send to Watch" buttons; treat these as visual placeholders to be replaced with sync-status pills before implementation.
12 ✓
Plan Builder — REMOVED. The coach is the planner. No "Build a Plan" wizard. Onboarding asks once for goals + current fitness + days/week. From there the coach generates the next 1-2 weeks adaptively and re-plans continuously. Adding a race triggers race mode automatically — the coach sees the new race on the calendar and quietly phases in marathon-specific work, scheduling the long-run progression and the taper backward from race day. No mode toggle. Recovery, weather, missed days all re-plan silently in real time; the coach explains why only when the user asks. Surface 17 (Plan Builder) is removed from scope. The "Training" surface (#2) is the canonical view of what the coach has generated — not a plan to "build," just what's planned for the next ~2-3 weeks with the full cycle arc visible if a race is targeted. The Plan Builder mockup is obsolete.
13 ✓
Coach Focus — a single thread that follows the runner across surfaces. When the coach is working on something specific with the runner — cadence drift, HR ceiling on long runs, cardiac-drift discipline, fueling timing, hill cadence — that "focus" is a first-class entity, not just a coach voice block. Pattern across surfaces:
  • Workout Detail (web + iOS): coach voice "FOCUS" block names the focus and the target ("Keep cadence above 180").
  • Watch Active Workout: live focus metric tile in xp-purple, with target and color-coded status — visible alongside pace verdict for the entire run. Hidden when no focus is active.
  • Run Recap: "did the focus hold?" badge alongside the pace reconciliation. Cadence avg vs. target, time-in-target percentage.
  • Insights: focus progress over time — "cadence avg: 174 → 181 over 8 weeks."
  • Coach chat: the coach references the active focus naturally without prompting.
Only one focus active at a time (no cognitive split). The coach picks the focus based on the user's data, current goals, and what they've been telling the user across the app. The focus is reflected in the data model: CoachFocus entity (metric, target, started_at, completed_at, narrative). The pattern means the coach has a single thread the user can follow — not a wall of generic insights.
14 ✓
Coach chat is action-taking, not Q&A. The chat is the primary surface for shaping the plan in natural language. Two flows that must work end-to-end:
  • Conversational plan modification. User says "There's an organized shakeout run for the race this Saturday I want to do. Add it and adjust accordingly." Coach replies with a proposed plan diff: shakeout added Sat 7am as 3mi @ E + strides, Sunday's long run shortened from 16 to 14, Wed threshold preserved. User taps Apply (or replies "go"), plan version increments, watch + calendar update via auto-sync. Audit trail logged: "Plan v23 → v24 · added shakeout, shifted long run, reason: race shakeout request 2026-04-21 09:14". Same pattern for: moving workouts, swapping types, adding strength sessions, deleting rest days, scheduling races mid-cycle. The coach proposes, the user confirms with one tap, the plan updates atomically.
  • Unplanned-run absorption. User goes rogue — trail run with a neighbor, an unscheduled easy run, an impromptu race. The watch records it, the activity syncs through HealthKit, reconciliation finds no matching prescribed workout. The coach auto-acknowledges on the next chat session and on Today: "Saw the trail run yesterday — 8.2mi, 1:14, easy effort. Counted toward this week's load. Tomorrow's threshold becomes general aerobic; you've earned the volume but you don't need both stresses on top of each other." The plan adapts automatically (the run happened — the plan can't not account for it). The user can label the run after the fact ("trail run, easy effort with Sam") via chat or the run detail; labels feed back into the coach's read of the user's training history. Override available: user can ask the coach to not count an unplanned run if it was a complete one-off.
Tools the coach calls: add_workout, modify_workout, delete_workout, add_calendar_event, shift_plan_block, regenerate_from_date, label_unplanned_run, recompute_load_after. All tool calls produce a plan version diff that the user reviews before applying. Authority model: the coach proposes; the user confirms (one tap from chat). Exception: same-day weather/AQI/heat shifts can auto-apply with notification (per Decision 08). Unplanned-run load adjustments auto-apply (the run already happened) with the coach narrating the adaptation.
Net effect on the plan

Simpler scope: Garmin/Coros integration dropped, race-photo integration dropped, manual subjective wellness dropped, plan-template picker dropped, "Send to Watch" CTA dropped, "Plan Builder" wizard dropped entirely. Sharper bets: Amp is the canonical strength surface, the coach is the planner (no wizards, race-add triggers race mode, re-planning is continuous and silent), recovery score is purely objective, sync is invisible. The product is more magical, fewer surfaces, and the build is meaningfully smaller. The pattern across all decisions: minimize manual user effort. The system does the work; the user runs.

Design notes (target made explicit + Coach Focus surfaced). Pace verdict at top: target range above the hero number; current pace huge, color-coded for status (green in range / yellow ±15s / red ±15s+); verdict line names the action in verbs ("Locked in" / "Pick it up" / "Ease back"). Coach Focus tile in xp-purple sits between pace and stats: shows whatever the coach is currently working on with the runner — cadence in this example, but also viable for HR ceiling, cardiac drift discipline, fueling timing, hill cadence, etc. Focus value color matches its own target (independent of pace status). When the coach has no active focus, the tile is hidden and the screen reverts to the simpler 3-element layout. The runner sees pace + focus simultaneously, both with explicit targets.
