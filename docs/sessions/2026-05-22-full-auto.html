Extensive audit — the top findings

Two deep agents reviewed every screen on web/iPhone/watch and every file in web/lib / web/app. Grouped here by severity. Items not yet fixed are ready for your review.

BLOCKER — leaks that contradict the "plain language, no citations" rule

  • VDOT leaked on screen in user-facing copy: web/app/profile/CoachReadsCard.tsx:218,257,354,385 ("Pace Bands · canonical Daniels for VDOT 52.1"); web/app/races/[slug]/page.tsx:525-667 ("Your VDOT 50.1 … goal needs VDOT 54.7"); web/app/(public)/landing/page.tsx:42,153; web/app/health/page.tsx:213,328,491. Should read "fitness score" everywhere.
  • HRR/Daniels framework jargon: web/app/health/HrAnchorsIsland.tsx:55 "Enables Karvonen (%HRR) zones"; web/app/overview/Z2CoverageCard.tsx:59 uses "Z2 / Z3 / Z4" tags; native/Faff/Faff/Screens.swift:507 exposes "HRR / %max" via the framework toggle copy; native/Faff/FaffWatch Watch App/IdleView.swift:113 shows "@ T · 6:31/mi" (Daniels code on the watch glance, where it's least decodable).
  • Pace migration banner (web/app/profile/PaceMigrationBanner.tsx:121-157) talks about "canonical Daniels values" and "VDOT 50.1" right in the banner.

SHARP — orange-as-today / amber-as-emphasis (the rule we agreed on)

  • iPhone: Faff.C.race orange is the tab-bar active color, the ADD/EDIT pill color, the race countdown ("· 12d"), and the StatPill accent (Components.swift:34,46,106,159,185). Orange should be reserved for warnings — convert active-tab, edit pill, and race countdown to ink/neutral.
  • iPhone: "Focus" coach verdict cards rendered amber (Screens.swift:342,1456,2017; WorkoutDetailView.swift:55) — "focus" is guidance, not warning.
  • Watch: IdleView.swift:25,58,71,78,140 uses the FAFF wordmark, pace line, and race-name in orange. Wordmark + pace label aren't warnings.
  • Web: training/page.tsx:340,386 race-week timeline + race-day bar value in var(--orange) — race day is a milestone, not a warning.
  • Web: log/RunDetailModal.tsx:115-116 "60% OF PLAN" and "BELOW PLAN" both tagged amber — should distinguish (we have SHORT vs DONE now; surface it here too).

SHARP — dead UX (clickable nothings, "coming soon" stubs)

  • web/app/workout/[date]/page.tsx:326-337 — "Send, coming soon" disabled button + "Weather wiring coming soon" tile in production. Cut both until they're real.
  • web/app/profile/ConnectorsCard.tsx:223-242 — eight inert "Coming soon" connector chips (Garmin, Coros, Polar, Suunto, Wahoo, Google Fit, Whoop, Oura, Final Surge, Training Peaks). Either gate to a "request" CTA or remove.
  • web/app/overview/WhyTooltip.tsx:20-101 — the "? Why this run" pill is tap-to-expand. Owner explicitly dislikes this pattern; the three lines (what / where in plan / point) should render inline under the hero, no tap.
  • web/app/races/[slug]/{GoalEditIsland,FuelEditIsland}.tsx — literal "tap to edit" copy. The row's already a button; drop the hint.

RISK — silent error swallowing where a runner gets confused

  • web/app/api/me/timezone/route.ts:52 — the timezone UPDATE silently catches DB failures. If it fails, every date calculation downstream is wrong with zero visibility. Log it.
  • web/app/api/strava/sync/route.ts:64 — per-activity loop swallows normalize errors. One bad activity silently wipes the sync. Record into strava_sync_state.errors.
  • web/app/log/page.tsx:74,90 — shoes + connectors reads return blank with no logging on DB failure.

RISK — multi-tenant landmines

  • Legacy 'me' defaults still in lib stores: profile-store.ts:22, profile-write.ts:20,65, goals-store.ts:31, plan-store.ts:174, prefs-store.ts:39,61, skip-store.ts:65,92,106. Any caller that forgets to pass userId writes to the orphan demo row. Required cleanup before multi-user beta. (The check-in path is now fixed; these stores are the remaining vector.)
  • iPhone Strava/HealthKit syncs always include user via Bearer auth → safe. But these lib defaults are the trap when adding any new server-side caller.

PARITY — web has features iPhone doesn't

  • Profile coach reads: web CoachReadsCard.tsx shows race feasibility, fitness contributors, all pace bands, HR zones, adaptive-VDOT banner. iPhone profile shows only three StatPills + shoes + integrations.
  • Z2 stimulus card: web Z2CoverageCard.tsx ships a full evidence / diagnosis / recommendation / falsifier finding on /overview. iPhone: nothing equivalent.
  • Pace migration banner, post-race awareness, Strava-gap card, adaptive-VDOT — all web-only.
  • iPhone run recap distance rounding shows whole numbers when round (OverviewModels.swift:740-743); web always 1-decimal. A 6.0-mi run reads "6.0" on web, "6" on iPhone.

DEAD code worth deleting

  • web/lib/seed.ts — already documented as a no-op stub, zero importers.
  • web/app/api/plan-week/route.ts, research/route.ts, elevation/route.ts, race-retrospect/route.ts — handlers with zero string-literal callers across the codebase. Confirm iOS doesn't hit them and delete.
  • web/app/components/v4/* — 14 components imported only by v4/index.ts, which itself has no importers. Decide: wire it into /overview or delete the folder.
  • web/app/components/ConfirmDialog.tsx — exported, no importers.
  • ~20 admin diagnostic API endpoints under web/app/api/admin/ with no UI entry point. Add a "Diagnostics" section on /admin that lists them, or retire the ones that are stale.

REAL TODOs that should be acted on

  • web/app/api/health/route.ts — 8+ TODOs for "stub each with coach.* call". The whole route is mockup-faithful placeholder data. The web Health page reads from health_samples directly (so the page works), but anything depending on this route gets perpetual stubs. Decision: wire or delete.
  • web/app/races/data.ts — 9 TODOs for missing race fields (PR list, B-classification, coach reads, trajectory, weather/location). Renders fake numbers on /races/[slug].
  • web/app/api/profile/route.ts:1122 — engine details stubbed.

POLISH — accessibility / dim text / unlabelled svgs

  • 19 inline <svg> components in web — only 2 have aria labels. The readiness ring, HR chart, elevation profile, course visual, route map are all unlabelled.
  • Dim text at rgba(8,8,8,.32) is below WCAG AA on white cards. Used in health/page.tsx, races/page.tsx, profile islands. Bump to .42 minimum.
  • Empty stat tiles render a bare "—" with no copy; add one-line "Connect Apple Health" or "Log a recent race" inline guidance.
