2 · What's working great

  • Tool-use coach with truth contract. Verified end-to-end: 7–9 tool calls per brief, voice anchored to today, threshold doctrine pulled from /Research/, no hallucinated runs.
  • Per-rep telemetry from watch → coach. Watch ships WatchCompletion Data, lands in coach_intents + strava_activities, coach's getWorkoutCompletion reads it. Synthetic test confirmed coach can cite per-rep pace consistency + HR drift + cadence lock.
  • Background regen. Mutations fire async warm. User mostly never waits.
  • Single source of truth across web, iPhone, watch, coach. Same DB, same endpoints, same tables.
  • Daniels-style structured workouts propagate cleanly from plan_workouts → web modal + iPhone card + watch payload — same shape everywhere.
  • /Research/ as the doctrine source. Coach's getDoctrine tool reads canonical markdown, no general-knowledge filling-in on quality days.
  • Lock + atomic counter is the right pattern for cross-agent ship arbitration.

5 · What we could add (proactive)

RECOMMEND · BUILD · P25

Runner notes on completed runs · your idea

Telemetry says you ran 6.2mi at 145bpm. Telemetry can not say the watch was glitchy, the heat was worse than forecast, the legs felt like spaghetti, or that you had a breakthrough mile you want to remember. The runner wants to talk about their run — the coach should listen.

Implementation sketch

  • Storage: add note: string to strava_activities.data jsonb (no migration). One text field per run, 0–500 chars.
  • UI: text field on the run detail modal (web + iPhone). Auto-saves on blur. Placeholder: "Anything the coach should know about this one? Quick notes welcome."
  • API: PATCH /api/runs/[id] accepts { note: string }.
  • Coach access: extend getRuns tool to include note per run. Coach absorbs naturally when discussing recent runs.
  • No cache bust on note add. Per your instinct — notes are context, not events. They flow through on the next natural regen (day rollover, next run logged).
  • Doctrine line in coach/prompts/index.ts post-run mode: "If the runner left a note (in getRuns output), absorb it into your framing. The note tells you what telemetry can't."

Effort: ~80 LOC web + ~50 LOC iPhone + 10 LOC tool extension + 3 LOC doctrine. Half a day total.

Bottom line

The architecture survives close inspection. Coach is honest, data flows from one source, mutations propagate, the daily cron warms the cache, and the watch ↔ iPhone ↔ web ↔ coach loop is real (verified end-to-end with synthetic post-run data).

The gaps are mostly iPhone-side surface coverage (/log, /runs/[id], /settings, /tips) and the v2 plan generator. None block what the app already does.

Runner-notes is the right next add — solves a real signal gap with low effort and parks the rest of the backlog gracefully.
