backend + security · 11 probes · 2026-06-05

Backend
audit

Eleven parallel probes across the Strava webhook, HK ingest, watch complete, coach state, readiness brief, plan adapter, onboarding, cron jobs, run writes, race projection, and the security perimeter.

Headline: this audit found ten P0 ship blockers. Three are remote code paths an attacker can hit today — Apple JWT signature isn’t verified, Strava OAuth state is the raw user_uuid in cleartext, and the Strava webhook accepts unsigned POSTs. The remaining seven are cross-user data leaks and silent-write losses sitting in the run, cache, race, and cron layers.

Cross-cutting patterns

01
Server-clock “today” survives the runner-tz refactor
The locked TZ doctrine (every “today” call uses runnerToday(userId)) was applied unevenly. 13 separate sites still compute today via new Date(Date.now() - 7 * 3600000), hardcoding PDT and ignoring DST. Hits readiness-brief, health-state, watch-complete, profile-state, plan-drift, adapt-block, build-watch, snapshot windows, four drift detectors, and the notifications cron. A NY runner gets the “race day 05:30” push at 22:30 the night before.
02
Conflict targets without user_uuid produce cross-user collisions
Six write paths (runs via Strava cron + webhook, races via onboarding + API, connector_tokens, coach_proposals) rely on global-uniqueness constraints with ON CONFLICT DO NOTHING/UPDATE targeting columns that don’t include user_uuid. When two users hash to the same id, the second write silently no-ops or, worse, overwrites the first user’s row.
03
Silent .catch(() => {}) swallows DB failures with no telemetry
Eight surfaces drop errors on the floor: last_sync_at updates, coach_intents writes, raiseAlert, pullSync post-write hooks, computeYesterdayPillars, four cron alert paths. A Postgres lock timeout produces a green 200 response and zero log signal. The “loud server-side write errors” doctrine codified in Task #69 was only applied to new paths, not retrofitted.
04
No request-perimeter defense in depth
Zero rate limiting on auth/ingest, zero CSRF tokens on cookie mutations, zero signature verification on Strava webhook POSTs, no admin gate on /api/admin/*. middleware.ts does not exist. Each layer alone is a P1; together they form the P0 attack surface that pairs with the three auth-bypass bugs.
05
Doctrine drift between sibling loaders
Three pairs diverge: loadStableBaseline exists verbatim in state-loader and glance-state; activeNiggle reads check_ins in one loader and the niggles table in the other; canonical.ts and pullSync.ts duplicate NEVER_COPY + shoe attribution + tier logic. A future schema change to any one source poisons only half the surfaces.
06
?? 0 and truthy-checks fabricate signal from absence
Same shape as the multi-tenant audit’s seed-adapter finding, now found in five more places: executionBufferSec = 30 default, rhrRow?.a truthy-checks, findCanonicalRow’s COALESCE-distance-to-zero, dynamic sleep target divorced from score, and the lone-HRV story asserting “sleep is holding” when there’s no sleep data. Each one prints a confident number off no real input.
07
Dead code and abandoned migrations linger after gut-jobs
deriveSplitsFromPhases, loadYesterdaySnapshot, detectRhrSpike, detectSleepCrater, refresh-briefings workflow, the direction ternary in computeMovers, the void goalPacePerMi line, and a manual-route migration that never landed (health_samples unique index). The 2026-05-28 LLM rip and the 2026-06-03 reactive-coach gut left tombstones the audit caught on second pass.

P0 ship blockers

P0 · auth bypass
Apple Sign-In JWT signature is never verified
app/api/auth/apple/route.ts:67-95
verifyAppleToken decodes the JWT and only checks iss, aud, exp. JWS signature verification is explicitly skipped in code with a comment admitting it. An attacker crafts a JWT with the right claims and any sub/email and signs in as any user; the email-bootstrap branch links forged identities to existing rows including David’s. Fix: install jose, fetch JWKS, jwtVerify with kid match. Until then gate the route on non-prod.
P0 · account takeover
Strava OAuth callback trusts state as user_uuid in cleartext
app/api/auth/strava/callback/route.ts:80-184
The callback decodes state straight into userId and writes the attacker’s Strava tokens into the victim’s connector_tokens row. No session check, no signature on state. Anyone who reads a victim’s user_uuid (URLs, logs, the connect-action response) can hijack their Strava link. Fix: require an authenticated session at callback and bind state to it via signed JWT or a server-side oauth_state table.
P0 · forged ingest
Strava webhook POST has no signature verification
app/api/strava/webhook/route.ts:65-153
subscription_id is read into a variable but never validated, no IP allowlist, no shared-secret path. An attacker POSTs { aspect_type: 'create', object_id, owner_id: victimAthleteId } and forces an authenticated activity fetch into the victim’s runs. aspect_type: 'delete' hard-DELETEs by id. Fix: validate subscription_id against the stored row, or mint a 64-byte secret path component for the webhook URL.
P0 · cross-user write
Cron INSERT into runs uses ON CONFLICT (id) DO NOTHING without user_uuid
lib/strava/pullSync.ts:447 (+ webhook /api/strava/webhook)
runs.id is the Strava activity id, primary-keyed alone. If user A already wrote that id (from any ingest path), user B’s cron insert silently no-ops — runner sees zero error, activity never lands. findCanonicalRow upstream is user-scoped, so the matcher said “new for this user” but insert collided with someone else’s row. Fix: add a unique index on (user_uuid, data->>'id'), surface foreign-owner collisions as hard errors.
P0 · cross-user write
Strava webhook DELETE FROM runs missing user_uuid filter
app/api/strava/webhook/route.ts:391-395
The pre-INSERT DELETE FROM runs WHERE id = $1::bigint on the upsert path scopes only by id, not user_uuid. If two Faff accounts ever linked to the same Strava athlete (or a future ingest path reuses Strava’s id space), a webhook delivered for one runner’s activity-update silently deletes another runner’s row. Fix: add AND user_uuid = $2 to the line-392 DELETE.
P0 · cross-user leak
race-header race meta lookup is not user-scoped
lib/coach/race-header.ts:341
When resolving the active plan’s race row for goalLabel + distanceMi + dateLabel, the query selects by slug alone. races.slug is per-user, not globally unique — two users with the same slug get the wrong row. Bleeds goal and distance into another runner’s RaceHeader. Fix: add AND user_uuid = $2; mirror the snapshot-projections cron path that has the same bug at route.ts:175.
P0 · cross-user cache leak
race-lookup Step-1 trusts planRaceId without user_uuid filter
web-v2/lib/coach/race-lookup.ts:59-62
loadNextARace queries SELECT slug, meta FROM races WHERE slug = $1 with no user_uuid clause. The 60s memoized cache then promotes whatever row was returned under a per-user key. The Step-2 fallback already passes userId — only Step-1 is broken. Same global-slug assumption as race-header above. Fix: add AND user_uuid = $2.
P0 · cross-user write
Onboarding race slug collision allows cross-user race overwrite
app/api/onboarding/complete/route.ts:395-411
Race slug is My ${distance}-${date} with no user component, upserted on global slug uniqueness. Two users picking “5K” on 2026-08-15 collide: second runner overwrites the first’s meta, and generatePlan then operates on the wrong user’s race. Fix: namespace the slug with userId, or make conflict target composite.
P0 · cross-user write
Manual-route ON CONFLICT relies on a unique index that no migration creates
app/api/health/manual/route.ts:87
UPSERT declares ON CONFLICT (user_id, sample_type, sample_date) but no migration creates that unique index. If missing, every manual entry 500s; if it exists keyed on user_uuid instead of user_id, it silently writes duplicates and never matches. Fix: ship the migration, verify it on prod before trusting this route.
P0 · TZ drift on cron
Notifications cron uses server clock for all per-user firing decisions
web-v2/app/api/cron/notifications/route.ts:193
listActiveUsers hardcodes tz_offset_min: 0 and nowInUserTz derives from UTC. Race-day morning, race-eve 21:00, weekly check-in, niggle/sick checks all fire at the wrong wall-clock. Compounds with three today.toISOString().slice(0,10) sites that tip the day key over at the wrong moment — race pushes silently never trigger after 16:00 PT. Fix: join profile.timezone, compute userNow via Intl.DateTimeFormat.

Security perimeter 9 findings
P0Apple Sign-In JWT signature is NOT verified — full impersonation
app/api/auth/apple/route.ts:67-95
verifyAppleToken decodes the JWT and only checks iss, aud, exp — JWS verification explicitly skipped (the comment block admits it). Email-bootstrap branch links the forged identity to ANY existing users row with that email, including David’s.
FixInstall jose, fetch JWKS at APPLE_JWKS_URL, pick key by header.kid, jwtVerify(token, key, { issuer, audience }). Until then gate route on NODE_ENV !== 'production'.
P0Strava OAuth callback trusts state as user_uuid in cleartext
app/api/auth/strava/callback/route.ts:80-184 + app/api/auth/strava/route.ts:109
No CSRF token, no signature, no nonce. An attacker who knows a target user_uuid completes Strava OAuth against their own account and the victim’s connector_tokens row receives the attacker’s tokens. Future syncs upload victim’s runs to attacker’s Strava, webhook deliveries for attacker’s activities land in victim’s runs.
FixSign state HMAC over {userUuid, nonce, expiry} with a server secret, OR write state into a server-side oauth_state table keyed by random opaque value. Verify in callback before persisting; require an authenticated session match.
P0Strava webhook POST has no signature verification
app/api/strava/webhook/route.ts:65-153
POST accepts any JSON whose owner_id maps to a registered athlete. subscription_id is read but never validated. Forged aspect_type: 'create' triggers an authenticated fetch; forged aspect_type: 'delete' hard-deletes runs by id.
FixValidate subscription_id matches strava_webhook_subscriptions row, or mint a 64-byte secret path component (/api/strava/webhook/<secret>) on subscribe and reject any POST without it.
P1No rate limiting on login, OAuth, or ingest surfaces
app/api/auth/email, app/api/auth/apple, app/api/auth/strava, app/api/ingest/workout, app/api/ingest/health
Zero hits for rateLimit in the repo. No middleware.ts. Email login’s only defense is the bcrypt cost factor (12) before pool exhaustion. Ingest endpoints can be hosed by a logged-in attacker to bloat runs/health_samples.
FixAdd middleware.ts keyed on (route, IP) using Upstash/Redis or in-memory LRU. 10/min/IP on auth, 30/min/IP on ingest.
P1No CSRF protection on cookie-auth mutations
lib/auth/session.ts:232 + all cookie-authed POST/PATCH/DELETE routes
extractToken accepts faff_session cookie or Bearer. SameSite=Lax allows top-level form-style POSTs from malicious sites. Zero CSRF tokens, zero Origin checks. Settings, plan, race, runs, profile all exposed.
FixReject cookie-auth requests where Origin != https://www.faff.run, or issue a CSRF token on session creation and require X-Csrf-Token. Bearer iPhone callers exempt.
P1/api/admin/* requires only any-authenticated-user, not admin
app/api/admin/strava-webhook/route.ts:35 + four sibling admin routes
requireUserId only verifies signed-in. Any user can subscribe/unsubscribe the Strava webhook for the entire app, backfill workout specs, re-enrich weather, audit coach intents. The header comment acknowledges this and says “when multi-user lands, replace.” Multi-user has landed; the migration was never done.
FixCentralize requireAdmin(req) that checks users.is_admin after requireUserId; apply to every /api/admin/* route.
P1Strava webhook DELETE drops user_uuid filter on the upsert pre-DELETE
app/api/strava/webhook/route.ts:222-228, 391-395
The explicit-delete branch at 222 IS user-scoped (good). The pre-INSERT DELETE at 391 is not. Combined with webhook forgery (no signature), attacker can force-delete victim runs by id alone.
FixAdd AND user_uuid = $2 to line-392 DELETE; gate the route on signature verification.
P2Session cookie not Secure in non-prod
app/api/auth/apple/route.ts:218, app/api/auth/email/route.ts:107
secure: process.env.NODE_ENV === 'production'. If Railway runs with non-prod NODE_ENV for any reason, the cookie ships without Secure. Session tokens are sha256-hashed (good) and lookups honor expires_at + revoked_at (good).
FixAlways-Secure (prod is the only env minting sessions), or move to sameSite: 'strict' with CSRF tokens (#5) closing the cross-origin gap.
P2/api/usage exposes global LLM spend without auth
app/api/usage/route.ts:13-21
GET returns per-day token/cost rollup across the whole coach_usage table — no auth, no scoping. Anyone hitting the URL sees total system spend.
FixGate behind requireAdmin or remove the route — comment block says it’s historical-only.
P2Email login distinguishes deactivated accounts from non-existent ones
app/api/auth/email/route.ts:57-66
Both "no such email" and "password mismatch" return invalid credentials (good), but suspended accounts return account not active — username-enumeration leak. Pairs with no rate limit (#4) for wider sweep.
FixCollapse both branches to invalid credentials / 401. Constant-time the bcrypt-compare path (no-password-hash row currently short-circuits).

P0Cron INSERT into runs uses ON CONFLICT (id) DO NOTHING without user_uuid in conflict target
lib/strava/pullSync.ts:447

P1Strava OAuth disconnect handler does NOT touch connector_tokens
app/api/auth/strava/route.ts:237-246
Disconnect revokes the token Strava-side and UPDATEs only profile.strava_*. connector_tokens — the documented source of truth — is never touched. After disconnect: getStravaToken still returns the revoked token, connection status reads ‘connected’, next cron pass picks them up again.
FixAdd UPDATE connector_tokens SET access_token = NULL, refresh_token = NULL, disconnected_at = NOW() WHERE user_uuid = $1 AND provider = 'strava' in the same transaction.

FixAdd per-user AbortController with 60s total budget; parallelize via Promise.allSettled in small batches or schedule per-user jobs separately.
P2connector_tokens UPSERT clobbers expires_at when Strava returns null
app/api/auth/strava/callback/route.ts:165
Stale future expires_at overwritten with NULL triggers per-request token refresh churn because getStravaToken reads expires_at as 0 when null.

P2getStravaToken race: two requests near expiry trigger parallel refreshes, second writes revoked token
