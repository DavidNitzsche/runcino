02The four layers

Faff sits on a single Railway Postgres. Every byte belongs to exactly one of four layers. Two are per-user (L1 + L3 output). Two are shared by every user (L2 doctrine + L4 ingest route code).

L1
Per-user data
Identity, physiology, runs, plan, races, niggles, injuries, sick episodes, check-ins, shoes, notifications, sessions, device tokens. Every row tagged with user_uuid. One row per runner per fact.
Isolated
~46 tables · keyed by user_uuid
L2
Research / doctrine
The coaching knowledge base — what the engine "knows." 28 canonical markdown files under /Research/, runner-facing summaries in learn_articles, curated race data in course_library, run templates in workout_library. The same data is visible to every user.
Shared · read-only
4 DB tables + 28 files
L3
Coach engine
Code (not data) that reads L1 + L2 and produces per-user output: briefings, plan adaptations, proposals, intents, audit logs. The engine is the same code for every runner; the output rows are tagged with user_uuid just like L1.
Shared code · per-user out
~6 output tables + lib/coach/*
L4
Ingest paths
How external data enters L1: Apple HealthKit (iOS bridge), Strava (OAuth + webhook), Apple Watch (completion payloads), manual runner input, coach engine derivations. One API route per source; all writes scoped to the calling session's user_uuid.

03L1 · The isolation contract

Every user has their own copy of every L1 fact. users is the identity anchor — its id column (a UUID) is the foreign key that every other L1 table uses as user_uuid. Nothing crosses between users.

L1 isolation rules

How user data is walled off

  • Every L1 table has a user_uuid uuid NOT NULL column with an FK to users(id)
  • Every read goes WHERE user_uuid = $1 — the $1 comes from the authenticated session
  • Every write/UPSERT carries user_uuid as part of its uniqueness key
  • Sessions resolve to exactly one user_uuid — Bearer (iOS) or cookie (web)
  • Connectors (Strava, HealthKit) attach their tokens to one user_uuid in connector_tokens
  • Strava webhooks route by external athlete-id → lookup the owning user_uuid → write to that user only
  • No row in L1 is global, shared, or "for everyone"
L2/L3 sharing rules

What every user inherits

  • The 28 canonical Research markdown files — read by the engine at runtime; same content for every user
  • 45 learn_articles rows — 21 SYSTEM DOCTRINE + 24 research summaries — readable by any signed-in user via /api/learn/[slug]
  • course_library — 4 hand-curated editorial rows (AFC, Big Sur, CIM, Sombrero) from legacy/web/data/courses/*.json + per-user stubs; real GPS attached later via /api/race/gpx or Strava match
  • The workout_library table — 54 doctrine-cited templates across 19 families (recovery, easy, medium-long, long variants, threshold, vo2max, speed, hills, fartlek, combo, marathon-specific, cutdown, ladder, race-specific, base-building, maintenance, walk-run, race, shakeout, rest) — every row carries a Research/ citation — engine reads at plan-build time via lib/plan/workout-library.ts with the old inline catalog as a safety-net fallback
  • The lib/coach/* + lib/plan/* code — pace zones, VDOT, ACWR, readiness, the 9 adaptation triggers — executed identically per runner
  • The briefing contract — surface × mode × topic schema — every runner gets the same payload shape
  • Onboarding doctrine — input tiers, fallback ladder, minimum-set — applies identically to every signup
The locked decision (2026-05-30)
No hybrid legacy/v2 coach engine. The best implementation of each subsystem becomes the rule of law, ported into web-v2/lib/coach/* with its citations intact. Every user — David, every future runner — runs against the same canonical doctrine. Personalization happens through inputs at L1, never through forks of the engine.

Every new user inherits the full body of doctrine at signup, zero migration required.

06End-to-end · one read, one write

To make the data path concrete: a runner finishes a tempo run on Strava on a 78°F morning. Here's exactly how that one event flows through the four layers — and where the per-user vs shared lines fall.

1
L4 · ingest
Strava webhook fires
POST /api/strava/webhook — payload arrives with the external Strava athlete-id.
Webhook is route code shared by every user. No L1 write yet.
2
L4 · ingest
Athlete → user_uuid resolution
SELECT user_uuid FROM connector_tokens WHERE provider='strava' AND external_id=$athleteId
L1 lookup. Resolves to exactly one runner. If no match → drop.
3
L1 · per-user write
UPSERT into strava_activities
Tagged user_uuid. Then UPSERT into runs & runs_processed for normalized fields.
Three rows, all scoped to that one user. Other users' tables untouched.
4
L4 · dedup
autoMergeForDate(userId, date)
Merges Strava-source row with any watch-completion row written earlier on the same date.
Per-user dedup. Source-of-truth precedence: actual_result > meta.finishTime > Strava elapsed.
5
L3 · engine read
Plan-state recompute kicks off
state-loader.ts reads the active plan (memoized via lib/plan/lookup.ts).
Engine code is shared. The query is WHERE user_uuid = $1. 60s TTL memoization.
6
L1 · per-user read
Read runner's recent runs + health
14-day RHR baseline, 2-night sleep, 7-day volume — all WHERE user_uuid = $1.
Pure L1. Other users' data invisible.
7
L2 · shared doctrine
Engine consults the doctrine
Reads Research/15-wearable-data.md (RHR thresholds), Research/01-pace-zones-vdot.md (zones), and Research/06-environment-and-weather.md (Maughan heat table).
Same files every user reads. Not parameterized; it's the rulebook.
8
L3 · engine compute
Adaptation triggers evaluated
All 9 triggers run against this one runner's data: missed_key_workout, rhr_spike, sleep_crater, volume_overshoot, niggle_reported, sick_episode_active, injury_active, pr_bank, goal_changed.
Same code path for every user. Personalization is the runner's L1 data, not branched logic.
9
L3 · engine write
Briefing + action log written
Row in briefings with the new (surface, mode, topics). Row in coach_actions as an audit record. Both tagged user_uuid.
Per-user output rows. Cache invalidated via bustPlanLookupCache(userId).
10
L1 · per-user read
Web/iOS pulls the new state
Next request to /api/today (or watch via iOS bridge) returns the briefing whose user_uuid = session.user_uuid.
Closes the loop. Same auth model for every client: Bearer or cookie → one user_uuid.
The pattern at a glance
L4 ingest writes L1 → L3 engine reads L1 + L2 → L3 engine writes more L1. Every L1 read is filtered by user_uuid. Every L2 read is unfiltered (shared). User A's data and User B's data never touch.

07Query patterns · the right way

The P0 incident this audit caught was a query pattern that mixed legacy identifiers — WHERE (user_uuid = $1 OR user_id = 'me'). The legacy 'me' clause matched David's pre-UUID rows for every new user. Cold-start users were getting David's plan. Fixed across 14 sites — now every L1 query is strict.

✗ Banned · the P0 leak (now gone)
SELECT id, race_id, mode
  FROM training_plans
 WHERE (user_uuid = $1 OR user_id = 'me')
   AND archived_iso IS NULL
 LIMIT 1;

Why bad: 'me' matched David's legacy row regardless of $1. Audited out of every loader on 2026-05-30.

✓ Canonical · strict user_uuid
SELECT id, race_id, mode
  FROM training_plans
 WHERE user_uuid = $1
   AND archived_iso IS NULL
 LIMIT 1;

Only the calling session's user matches. Verified 22/22 cold-start checks.

A read that joins L1 (user) with L2 (doctrine)
-- One runner's adaptation check pulls 14d RHR (L1) + threshold doctrine (L2)
-- Engine code lives in lib/plan/adapt.ts → detectRhrSpike()

-- L1 · user-scoped
SELECT AVG(value_num) AS rhr_3d
  FROM health_samples
 WHERE user_uuid = $1
   AND metric = 'resting_hr'
   AND sample_date >= CURRENT_DATE - INTERVAL '3 days';

-- L2 · shared doctrine (the threshold lives in Research/15-wearable-data.md
-- and is mirrored into learn_articles for in-app reading)
SELECT body
  FROM learn_articles
 WHERE slug = 'health-watch-thresholds';
 -- → "RHR >7 bpm above 14d baseline for 3 days → downgrade next quality"
A write that respects the per-user contract
-- Adaptation trigger writes a coach proposal — always with user_uuid
INSERT INTO coach_proposals
       (user_uuid, proposal_type, payload, status, created_iso)
VALUES ($1, 'rhr_spike_downgrade', $2::jsonb, 'pending', NOW());

-- And the matching audit row in coach_actions — also user_uuid
INSERT INTO coach_actions
       (user_uuid, action_kind, citation, payload, created_iso)
VALUES ($1, 'detect.rhr_spike',
        'Research/15-wearable-data.md §Resting-HR-deltas',
        $2::jsonb, NOW());

09Onboarding · what L1 rows get planted

When a new user finishes the 7-step onboarding flow, exactly the following L1 writes happen — atomically, in a single transaction. Zero L2 or L3 code changes. Zero migrations. Just rows.

1
L4 · auth
Magic-link sign-in creates the identity
users row (id = new UUID, email, name) + sessions row (Bearer/cookie token).
The new user_uuid is born here. Every subsequent row threads it.
2
L1 · profile
Profile + prefs written in one BEGIN/COMMIT
profile (birthday, sex, height_cm, experience_level, units, RHR/LTHR/maxHR if known) + user_prefs (preferred run days, default time, timezone).
Atomic transaction · ROLLBACK on error · no half-onboarded state possible.
3
L4 · connectors (optional)
Strava / Apple Health if granted
OAuth tokens land in connector_tokens. If skipped, fallback to manual-entry paths.
Apple-Health-optional · web-only users get /api/health/manual.
4
L1 · race / goal
First race or personal goal pinned
races row with priority='A' (default · changed from 'C' on 2026-05-30) OR personal_goals row.
Race-priority doctrine = A is the goal race. Anchors the plan generator.
5
L3 · auto-generate
Seed plan built by the shared engine
generatePlan() reads the new user's L1 (experience_level, target race date, VDOT or beginner-default) and the shared L2 doctrine (distance-specific block structure, volume floors, quality mixes). Writes training_plans + plan_weeks + plan_phases + plan_workouts.
Same engine that built David's plan. Personalization = the new user's L1 inputs.
