READ THIS FIRST · THE FOUR LAYERS

Faff's backend is four layers stacked on one Postgres database. Every piece of information in the system belongs to exactly one layer. Every agent — web, iOS, Claude — should know which layer it is reading from and writing to.

The coach is briefing-driven. The engine generates structured briefings per (surface, mode); the runner replies via typed reply chips on briefing cards. Doctrine lives as markdown files under /Research/ and is read by the engine at runtime; the runner-facing learn_articles table surfaces summaries back to the in-app reader. The brand voice is "direct" — honest, time-aware, never hype.

One database, two clients, one auth model. Railway Postgres holds everything. web-v2/ exposes 56 REST routes. Both faff.run web and the iOS TestFlight app consume those same routes via opaque session tokens (Bearer or cookie). There is no separate "API server" or "iOS backend."

This doc is the contract. When a new user signs up, exactly the same L1 row template fills in. The L2 research layer is shared and grows centrally. The L3 coach layer fires automatically off the runner's L1 + L2 reads. The L4 ingest paths plug in new providers (Garmin, Coros, Polar, Whoop, Oura — schema is already pre-allowlisted) without schema changes.

2 · L1 — Identity & auth — the canonical per-user model

This section defines the row template that EVERY user — David today, any future sign-up — fills in. The schema is identical; only the values differ.

Canonical answer · For every agent: users holds auth/identity primary keys (id, email, name, admin status, timezone). profile holds the rich runner data (LTHR, height, experience level, fueling, notification prefs, Strava OAuth tokens). user_prefs holds the weekly training schedule (long-run dow, quality dows, rest dow). All three are keyed by user_uuid in the v2 read path. The legacy user_id='me' column on profile + user_prefs is the single-tenant fallback for David's account only and is being phased out — new sign-ups never write 'me'.

3 · L4 — Ingest paths — where data comes from

Every byte of runner data enters the system through one of five paths. Each path has a single owning API route, a clear authentication contract, and a defined destination table in L1. Every signed-up user (David today, any future sign-up) uses these same five paths.

Apple
HealthKit
Path: iOS app reads HealthKit on a schedule + on observer fires → POST /api/ingest/health with Bearer.
Auth: Bearer (session kind=app).
Writes to:
  • health_samples — every sample (UPSERT on (user_id, sample_type, sample_date)). 19 sample types supported today: hrv, resting_hr, max_hr, respiratory_rate, sleep_hours, wrist_temp, run_power, cadence, vertical_ratio, ground_contact_time, stride_length, vertical_oscillation, spo2, active_energy, vo2_max, body_fat_pct, lean_mass, body_mass, hr_recovery.
  • users.max_hr / users.resting_hr — auto-ratchet from new max_hr / resting_hr samples (unless *_override is set).
  • profile.vo2max_apple — wellness signal only.
  • profile.health_connected_at — bumped on first ingest.
  • connector_tokens(provider='apple_health') — last_sync_at, activities_count.
Cadence: iOS bridge runs on background fetch + HealthKit observer fires + app launch. Idempotent UPSERTs.
David today: 2,720 samples, 19 types, range 2025-05-22 → 2026-05-30.
Strava
Paths (3):
  1. OAuth connect · GET /api/auth/strava → upserts users (if new), writes profile.strava_access_token + refresh + expires + strava_athlete_id, writes/updates connector_tokens(provider='strava').
  2. Activity sync · cron /api/cron/keep-warm + on-demand from web. Calls Strava API with refreshed token, normalizes each activity, UPSERTs strava_activities (PK = Strava activity id). Tombstones go to deleted_activity_ids.
  3. Webhook · Strava POSTs to /api/strava/webhook on activity create/update/delete. Logged to strava_webhook_events; triggers a sync of just that activity.
Auth: OAuth code (sign-in flow). Webhooks use a shared verify token.
Writes to:
  • strava_activities · normalized payload in data JSONB; full Strava response in detail JSONB. Auto-attached shoe via shoe_id.
  • strava_pushes · audit when strava_writeback renames the activity.
  • strava_sync_state · cursor (currently global key/value).
  • strava_webhook_events · raw webhook log.
David today: 125 activities, connected 2026-05-18, latest sync 2026-05-25.
Apple
Watch
Path: Watch executes a workout from the prescribed plan → on completion, iOS HealthKit observer reads HKWorkout + HKWorkoutRoute → POST /api/watch/workouts/complete + POST /api/watch/route.
Auth: Bearer.
Writes to:
  • workout_completions · prescribed-vs-executed per-interval (phases JSONB). UPSERT on (user_id, workout_id).
  • workout_routes · polyline + per-mile splits. UPSERT on (user_id, started_at). Powers the map for watch-only runs that never reach Strava.
  • health_samples · workout HR samples flow through the HealthKit ingest path.
Read by: /runs/[id] for the recap; /api/watch/today to send back the next prescribed workout.
David today: 10 workout_completions, 24 workout_routes.
Manual
(runner)
Paths: One API route per kind. All Bearer-authenticated.
  • POST /api/checkin → check_ins (v2 reply-chip flow, post-briefing rating + note + extras) + daily_checkin (legacy energy/soreness/stress 1-10).
  • POST /api/niggle → niggles (body part + severity 1-10 + status). Recovery selection writes niggle_recovery.
  • POST /api/sick → sick_episodes (symptoms[], has_fever, started). Recovery writes sick_recovery.
  • POST /api/run/manual → strava_activities with a synthetic id (for watch-only or untracked runs that user logs by hand).
  • POST /api/shoe → shoes (brand, model, color, run_types[], mileage_cap). Mileage auto-increments on run attribution.
  • POST /api/race → races (slug, meta JSONB, course_source). Auto-matches course_library by slug.
  • POST /api/profile + POST /api/profile/notifications + POST /api/settings → profile + users + user_prefs.
  • POST /api/today/skip → skipped_workouts.
  • POST /api/coach/proposal accept/reject → coach_proposals.status.
  • (not yet exposed) Runner free-text notes → runner_notes (schema ready, no UI yet).
  • (not yet exposed) Strength sessions, cross-training sessions, recovery sessions → respective tables (schema ready).

Coach
engine
Path: Triggered by user actions (open /today, ack intent, accept proposal) or cron (refresh-briefings, run-adaptations). Reads doctrine from /Research/*.md + the runner's data, calls Anthropic API, writes derived rows.
Writes to:
  • briefings · generated payload (surface, mode, signature, payload JSONB).
  • coach_actions · audit of every adapt/notify/propose (mode: unilateral/propose/notify).
  • coach_proposals · runner-facing accept/reject items.
  • coach_intents · closed-loop commitments ("Lock in 168 cadence"). UI surfaces with reply chips.
  • coach_reads_cache · TTL'd per-runner cache of computed reads. Invalidated on relevant input changes.
  • coach_today_cache · global signature cache for /today briefings.
  • coach_usage · per-call telemetry (input/output/cache tokens, cost_micro_usd).
  • plan_mutations · audit row when the adapter changes a plan_workout.
  • plan_workouts field updates (the adapter mutates rows in place).

5c · The contract

  • One response shape per route. Web and iOS both consume the same JSON. Clients render differently; the server never branches on User-Agent or client kind.
  • Bearer or cookie, never both. userIdFromRequest() in lib/auth/session.ts resolves one or the other. iOS = Bearer (sessions.kind='app'). Web = cookie (sessions.kind='cookie'). Both are 32-byte hex tokens, sha256-hashed at rest, 90-day TTL.
  • Watch is a presentation layer. It never holds the source of truth; it never auths to the API. The iOS bridge is the data path.
  • Doctrine reads go through /api/learn/[slug]. Both web + iOS read system-doctrine rows the same way. The watch never displays doctrine — it shows numbers and intervals only.

Truth contract — prereqs gate topics

Rules for adding new doctrine

  • Append-only within the session. Locked rules are never silently mutated. When a rule turns out to be wrong, explicitly revise with a citation linking old → new.
  • Cite the source. Every doctrine row has citations_json pointing to (a) the markdown rule (CLAUDE.md section or Research file), (b) the code module that enforces it, OR (c) the migration that locked it.
  • Three places to write. Locked behaviors land in (i) CLAUDE.md for project memory, (ii) code (with the inline-comment citation), (iii) learn_articles with eyebrow='SYSTEM DOCTRINE'. The Research files are upstream of all three — they're the input doctrine that gets distilled.
  • Locking is deliberate. A rule is "locked" when it's been validated and the team agrees it's the canonical answer. Pre-lock rules are candidates — they live in conversation or PR descriptions, not in any of the three places above.

19 · Rules for agents reading this

  1. Read this doc before writing schema or new ingest paths. The canonical storage map (section 4) tells you which table a piece of info belongs to. Pick that one.
  2. Never write user_id='me' for new users. That value is reserved for the legacy David row. New users get a UUID-only row.
  3. Always scope per-user reads by user_uuid = $1. The legacy fallback OR user_uuid IS NULL exists in older code paths only — do not add new ones. If you find one, the fix is to backfill the row, not extend the fallback.
  4. L1 is per-user. L2 is shared. L3 is per-user but server-only. Never let an L1 surface (UI form, mobile page) write directly to an L2 table. Never let a runner write to an L3 table from the client — proposals/intents come back via documented routes that the coach engine reconciles.
  5. Doctrine lives in files, derived state lives in DB. Never insert into learn_articles from manual edits at runtime; updates should come from a deliberate seed run after Research changes.
  6. UPSERT, never duplicate. Every per-user table has a logical uniqueness constraint (e.g. health_samples (user_id, sample_type, sample_date)); use ON CONFLICT.
  7. Manual user input writes are idempotent. Re-posting the same check-in / niggle / RPE should update, not duplicate.
  8. iOS and web both hit the same routes. Don't fork. Authentication kind (cookie vs Bearer) is the only difference, and userIdFromRequest() abstracts it.
  9. Coach engine reads are cached aggressively. Hit coach_reads_cache first; invalidate explicitly when inputs change.
  10. Strava writeback is opt-in via users.strava_writeback. Don't push back to Strava without it.
  11. Never drop columns without a migration in web-v2/db/migrations/. Schema changes need review before they land. Reads can be updated independently.
