## iOS deploy

The Theme.swift extension + Info.plist `UIAppFonts` are in. **You need to bundle the Oswald + Inter TTFs into the Xcode project before TestFlight will show the new typography.** Without TTFs, the app falls back to system bold — clean degrade, no crash.

### 1. Get the TTFs

Download from Google Fonts (SIL OFL — open source, no licence cost):

- **Oswald**: https://fonts.google.com/specimen/Oswald → "Get font" → Download family
  - Need: `Oswald-Regular.ttf`, `Oswald-Medium.ttf`, `Oswald-Bold.ttf`
- **Inter**: https://fonts.google.com/specimen/Inter → "Get font" → Download family
  - Need: `Inter-Regular.ttf`, `Inter-Medium.ttf`, `Inter-SemiBold.ttf`, `Inter-Bold.ttf`

If the Google Fonts download bundles variable fonts (`Inter[opsz,wght].ttf`), use the "Static" subfolder where each weight is a separate file.

### 2. Add to the Xcode project

```
What:    Bundle 7 TTFs as project resources
Where:   Xcode → native-v2/Faff project
Exact:   1. In Finder, drop the 7 .ttf files into:
            Runcino/native-v2/Faff/Faff/Resources/Fonts/
            (create the Fonts/ subfolder if it doesn't exist)
         2. In Xcode → project navigator → right-click "Faff" folder
            → "Add Files to Faff..."
         3. Select the Fonts/ folder
         4. Check "Add to target: Faff"
         5. Click Add
Verify:  Build Phases → Copy Bundle Resources should list all 7 TTFs.
         Run a debug build · the Poster/Theme.Font.display calls
         should render in Oswald, not system bold.
```

### 3. Bump build number + archive + upload

Standard ritual per OPERATIONS.md §3:

```
What:    Ship v3 design build to TestFlight
Where:   Xcode → native-v2/Faff.xcodeproj
Exact:   1. Project settings → General → bump Build by 1
         2. Product → Archive (~3 min)
         3. Organizer → Distribute → App Store Connect → Upload
         4. App Store Connect → TestFlight → wait for processing (~10 min)
         5. Internal Testing → click the build → add to your tester group
Verify:  Open faff on your phone — Today surface should show Oswald
         verbs. Health write/read still works (no usage-string changes).
```

### 4. iOS rollback

If the build crashes or the Oswald lift introduces a regression:

```
What:    Roll back to a prior TestFlight build
Where:   App Store Connect → TestFlight → Internal Testing
Exact:   Add the previous green build to the tester group; the new bad
         build can stay in the build list (it just isn't being tested).
Verify:  Reinstall the app on your phone (or wait for TestFlight to
         push the older build, ~5 min). Crash should be gone.
```

## Working with the Faff design system going forward

**Source of truth lives in the Faff repo** (`/Volumes/WP/06 Claude Code/Faff/`). Always edit there first:

| Change | Edit in Faff | Then mirror to Runcino |
|---|---|---|
| Token (color, gradient, font) | `Faff/design/tokens/*.css` + `Faff/shared/tokens.json` | `Runcino/web-v2/app/globals.css` + `Runcino/native-v2/Faff/Faff/Theme.swift` |
| Component (Poster, Sibling, etc.) | `Faff/apps/web/src/components/<Name>.{tsx,module.css}` | `Runcino/web-v2/components/faff/<Name>.{tsx,module.css}` |
| Type contract | `Faff/shared/types.ts` | `Runcino/web-v2/lib/faff/types.ts` |
| Adapter logic (Glance → Faff payloads) | n/a — this is Runcino-specific | `Runcino/web-v2/lib/faff/glance-adapter.ts` |

A `scripts/sync-faff-to-runcino.sh` mirror script is a future cleanup task. For now, the mirror is manual — edit Faff first, then `cp -v` into Runcino, then test.

### Why this split?

`Faff/` is project HQ (design specs, research, agent system, fixtures). `Runcino/web-v2/` is the deploy target (Railway builds from it). Long-term goal is to fully migrate the codebase to `Faff/apps/web/` and deprecate `Runcino/`, but that's a methodical N+1 cycle — not this cutover. The short-term split respects Cardinal Rule #1 (build it right · don't break what works).

## Cutover decision log

- **Strategy: lift design INTO production** (vs. lift backend INTO Faff). Chosen because Railway already deploys `web-v2/` cleanly with all env vars set; reconfiguring Railway adds risk with no visible win. (Session 2026-05-28.)
- **iOS scope: Theme + Info.plist only** (vs. replacing live Views with new Surfaces). Chosen because replacing the live iOS app needs Xcode work + risk of breaking working Today/Watch flows. Visual upgrade happens when TTFs are bundled — code is ready to consume them.
- **Other web surfaces (/plan etc.) untouched.** They auto-inherit Oswald via the token swap; visual is consistent. Full rewrites are next cutover.
