02Move 1 · State-presence module + cold-start sweep

Pattern 2 + Pattern 8
Cold-start gate · single source of truth
Pattern 2 (cold-start envelope), Pattern 8 (5 disagreeing presence predicates)
The problem

Five different files defined "do we have real RHR signal?" differently · some required current + baseline, some required 5+ samples, some required just non-null. A Strava-only web runner landed at score 67-75 = READY because LOAD fired alone and no consumer agreed on what "we have nothing" meant.

The fix

A new lib/coach/state-presence.ts module defines hasSignal(), hasRecoverySignal(), recoveryCoverage(), and pillar-specific helpers (hasSleepSignal, hasHrvSignal, …). Recovery pillars (sleep / hrv / rhr / hr_recovery) are defined as the canonical group · LOAD does NOT count. Cold-start sweep then re-wired four consumers to use it: recovery-brief, readiness-brief.breakdownIsEmpty, health-actions, and heat-acclimatization.

03Move 2 · Web-side TZ capture

Pattern 3
Strava-only web users stop being stuck in UTC
Pattern 3 (web users never trigger captureTimezoneFromDevice)
The problem

captureTimezoneFromDevice() only fires on watch + iPhone HK ingest. A web-only Strava runner who never opens the iPhone app had no path to populate profile.timezone, so every server-side runnerToday() fell back to UTC · "today" off by up to 7 hours, cold-start recovery panels disappeared at UTC-midnight (5pm Pacific), readiness "today" queries returned null on real evenings.

The fix

New POST /api/profile/timezone wraps the existing idempotent capture helper. The web client pings it from two surfaces: (1) Shell.tsx on every authed page mount (sessionStorage-gated so it fires once per session), (2) StravaConnectionCard.startReconnect() before the OAuth redirect (defense-in-depth for users who land on /onboarding pre-Shell). Onboarding's Step3Confirm already captures TZ in the body · this is the legacy-user / dev-seed / failed-write safety net.

04Move 3 · noData gates on five fabricating tiles

Pattern 1
Fake zeros under green chips · five tiles fixed
Pattern 1 (?? 0 / || default literal fallbacks)
The problem

The seed adapter's adaptHealth() coerced missing health values to literal zeros via ?? 0, but only set noData: true on six pillars. WRIST TEMP, RESP RATE, SPO₂, BODY FAT, LEAN MASS · these five hardcoded noData=false, so an unconnected-source runner saw "0 °C", "0 /min", "0 %" with a green "on baseline" chip. SpO₂ was the worst · its >=96 ? 'good' check made fake zero render as "good" with a 0 % headline.

The fix

Added hasWristTemp / hasRespRate / hasSpo2 / hasBodyFat / hasLeanMass / hasMaxHr / hasActiveEnergyToday / hasActiveEnergyAvg7 presence predicates in adaptHealth(). Each tile now threads !hasX into both its status ('neutral' instead of 'good') and its noData arg, so the renderer shows an em-dash and the "no data yet" chart band instead of fake bars.

05Move 4 · Data-shape persona bench + CI gate

Pattern 8
Cold-start envelope locked into vitest
Pattern 8 (no test coverage on non-David data shapes)
The problem

The plan-engine bench (plan-engine.test.ts) covers PLAN shapes · peak weekly miles, VDOT trajectories. Nothing covered DATA shapes · what CoachState looks like for a Strava-only runner, a watch-only runner, a manual logger. Any commit could silently rewire so a cold-start runner gets a confident readiness score and no test would fire.

The fix

New lib/coach/data-shape-personas.test.ts defines four synthetic CoachState payloads (strava-only-web, watch-only-iphone, manual-only, mixed-partial) and runs each through state-presence + recoveryCoverage + summarizePresence. The CI gate is the explicit test "never lets a Strava-only runner present as READY with full recovery" that locks hasRecoverySignal(strava-only) === false · if any future change breaks that, the suite fails.

06Move 5 · Pillar-claim honesty in prose

Pattern 5
Three prose lines stop lying
Pattern 5 (cited prose makes claims without checking the citation)
The problem

Engine prose made "all five pillars" / "all systems" / "multiple pillars" claims without counting how many were actually reporting. A Strava-only web runner with just LOAD signal still got a 'ready' band score and saw "All systems in their normal band" · a lie when 4 of 5 pillars had no observation at all.

The fix

Three lines now read breakdown.inputs filtered to real observations (observedV not in 'no data' / 'building history', weight !== 0) before composing:

  • synthesis.buildSharpStory · "All N pillars" / "M of N pillars" / "the one pillar reporting" / "Score is in the sharp band" instead of the hardcoded "all five."
  • readiness-brief.buildHeadline · the ready / moderate / pull back branches all read real pillar count + name the dipping or flagging pillar by label when there's exactly one.
  • Heat-acclim's rhrTrend gate (4fcf4391) was the first cite-or-shut-up fix · this finishes the sweep for the two most user-visible top-line strings on the morning brief.
