The hero number question · DEFENDED. Round 1 used target pace as hero. Wrong. During execution the runner is working against current pace — that's what they're trying to adjust. Target is the prescription already known going in. Make current dominant, target a thin reference strip. Delta chip color carries the "you're slow vs fast" read in 0.5 seconds — runner never has to parse "+15s" as text, just sees red/green/amber.

The screen isn't one design — it's one pattern that adapts to workout context. Three variants below show the swap rule: which metric becomes hero depends on whether pace or HR is the governing prescription.

Variant 1 design defense · threshold/interval

Hero choice: current pace at 100pt · the prescription says 6:31 but execution is "am I hitting it" · current is the actionable metric.
Target as reference strip: 24pt at top · runner already knows it going into the rep · displayed for trust ("watch hasn't drifted from plan"), not for primary reading.
Delta chip color logic:
  • green chip "−2s" = you're 0-5s/mi faster than target (good, on threshold)
  • green chip "+0s" = exactly on target
  • green chip "+1s to +5s" = within tolerance (the workout doesn't require pixel precision)
  • amber chip "+6s to +14s" = drifting (cue: "stay on it") · pace number color stays orange
  • red chip "+15s+" = sustained drift (cue: "well off") · pace number turns red
  • amber chip "−6s+" = pushing too fast (cue: "ease back") · pace number stays orange
HR is footer-only: 168 BPM shown but small · pace is the prescription on threshold, HR is supporting context (informs whether the runner is gassed early).
Cadence (SPM): third footer stat · useful for form drift detection late in a workout but not primary.
When this variant fires: workout.type ∈ {threshold, interval, tempo, repetition} OR explicit pace target on the rep

Variant 2 design defense · easy / Z2 stimulus

Hero choice: HR at 100pt · V5 explicitly says HR is the governing metric on easy days · pace is whatever cardiovascular ease produces, not what the runner targets.
Ceiling not target: Z2 has a ceiling (139), not a target. Below ceiling = stimulus is correct. Above ceiling = drifting out of zone. The strip says "CEILING" not "TARGET" so the semantic is unambiguous.
Color logic on HR hero:
  • green = HR is below ceiling (Z2 stimulus achieved)
  • amber at ceiling-5 to ceiling-0 = "ease back" cue
  • red above ceiling = sustained Z2 violation · same haptic escalation as pace drift
Pace demoted to footer: 8:42 visible but small · runner sees what their easy pace is producing today without making it the prescription.
Why green for the phase chip: easy days are recovery-adjacent · the same green used on the recovery / readiness markers.
When this variant fires: workout.type ∈ {easy, recovery_run, long_run_aerobic} · explicit HR ceiling exists on the rep · no pace target set

Variant 3 design defense · race-pace effort

Hero choice: current pace at 100pt · race-pace is a pace prescription · same structure as threshold but with an HR-ceiling watchdog layered on.
HR-ceiling chip (top-left): "HR 175 / 178" · shows actual / max-permitted on the rep · amber background once HR enters the warning band (within 5 BPM of ceiling) · turns red when HR breaches ceiling. Runner sees "I'm at pace but redlining cardio" in one glance.
HR also shown amber in footer: footer BPM stat color-codes too (175 in amber) · redundant signal protects against sweat-on-screen + missed chip.
Why this matters: race-pace efforts at the wrong CV cost waste training adaptation. Workout asks for race-pace, but if HR is screaming you'll blow up the next rep · the chip is the early warning that pace alone wouldn't surface.
Color escalation on HR:
  • neutral = below 95% of max-permitted (safe band)
  • amber = 95-100% of ceiling (caution, you're at the edge)
  • red = above ceiling (you're racing the rep, not running it)
When this variant fires: workout.type ∈ {race_pace, marathon_pace, half_pace, simulation} OR (explicit pace target AND explicit HR ceiling on the same rep)
Hero-number swap rule · single decision tree
if workout.rep.pace_target AND NOT workout.rep.hr_ceiling:
    hero = current_pace            # Variant 1 (threshold/interval)
    reference_strip = target_pace
    footer_includes_hr = true (small)

elif workout.rep.hr_ceiling AND NOT workout.rep.pace_target:
    hero = current_hr              # Variant 2 (easy/Z2)
    reference_strip = hr_ceiling   # labeled "CEILING" not "TARGET"
    footer_includes_pace = true (small)

elif workout.rep.pace_target AND workout.rep.hr_ceiling:
    hero = current_pace            # Variant 3 (race-pace)
    reference_strip = target_pace
    overlay_chip_hr_ceiling = true # shown top-left of hero
    footer_includes_hr_with_color_escalation = true
Layout invariants across all three variants (preserves muscle-memory across workout types):
  • Hero metric always center · 100pt SF Compact Display 800-weight · color = state
  • Reference strip always top · 24pt · labeled "TARGET" or "CEILING"
  • Delta chip always top-right · color carries the read
  • HR-ceiling chip (only Variant 3) top-left · color escalation distinct from delta
  • Footer always three stats · the third stat (cadence) is the only constant across variants
What changes between variants:
  • Phase chip color (orange threshold · green easy · orange race)
  • Hero metric (pace vs HR)
  • Reference strip label ("TARGET" vs "CEILING")
  • Footer stat ordering (Variant 1: time/HR/cadence · Variant 2: pace/distance/cadence · Variant 3: HR/time/cadence)
  • Variant 3 adds the HR-ceiling chip overlay
This is one screen pattern · three configurations · driven by what the workout prescribed.
