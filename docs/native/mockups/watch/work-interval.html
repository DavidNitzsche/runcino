WHAT CHANGED FROM V2

V2 used one big hero (100pt) with everything else demoted. Wrong — running needs to track multiple metrics. Apple's native workout app shows 4-5 metrics simultaneously, each at 32-44pt logical (≈ 64-88pt mockup), each color-coded by purpose.

V3 keeps Faff's brand identity (Bebas Neue italic ALL CAPS, gradient orange) while adopting Apple's proven multi-metric layout. Three metrics stacked vertically, each with a tiny italic ALL CAPS label and a big Bebas numeral. Reading each metric takes < 0.3s because the type is huge and the labels are positioned consistently.

VARIANT 1 · THRESHOLD

Layout: three big metrics stacked vertically (PACE 64pt Bebas · HR 64pt · REP TIMER 64pt) plus a 3-stat footer ribbon (target / cadence / distance, all Bebas italic 13pt).

Color = state, applied to the metric that's STATE-CARRYING: pace lives green in range, amber drifting, red well-off. HR stays white because the prescription doesn't gate on HR for threshold. REP TIMER stays brand-orange so the runner always knows where they are in the rep without parsing color.

Delta chip: "−2s" lives INSIDE the pace row as an inline chip with solid bg + black text. At 22pt Bebas italic it pops against the dark screen — same chip color as the pace numeral so the read is unified.
FIRES WHEN: workout.rep.pace_target AND NOT workout.rep.hr_ceiling

VARIANT 2 · EASY · Z2

HR moves to position 1 because it's the prescription — V5 doctrine: HR is the governing metric on easy days; pace is whatever cardiovascular ease produces.

Color migrates to HR row: green below ceiling (133, Z2 stimulus achieved) · amber at ceiling (138, "ease back" cue) · red above ceiling (147, drift out of zone).

Phase chip turns green ("EASY · Z2") · easy days are recovery-adjacent · reuses the recovery green so the runner reads "this is restorative" without parsing.

Distance gets the third row (instead of REP TIMER) · easy runs are continuous, not rep-structured · what the runner wants to know is "how far have I gone."
FIRES WHEN: workout.rep.hr_ceiling AND NOT workout.rep.pace_target

VARIANT 3 · RACE-PACE

Layout matches V1 (pace dominant) — but the HR row label dynamically changes: "HR · Z4" → "HR · NEAR CAP" → "HR · OVER CAP". HR numeral takes on amber/red color escalation.

Same color logic as V1's pace, applied to HR: near-cap (within 5 of ceiling) = amber, over = red.

Why this matters: race-pace efforts at the wrong CV cost waste training adaptation. The runner can be on pace AND redlining — the chip says "ease pace next rep even though target says 7:01."
FIRES WHEN: workout.rep.pace_target AND workout.rep.hr_ceiling
LAYOUT INVARIANTS · MUSCLE MEMORY ACROSS VARIANTS
  • Header is ALWAYS: phase chip (Bebas italic, ALL CAPS, color = phase) + distance/total + thin progress line
  • Position 1 metric = the prescription (pace on threshold/race-pace, HR on easy)
  • Position 2 metric = the supporting CV signal (HR on threshold/race-pace, pace on easy)
  • Position 3 metric = the temporal anchor (rep timer when rep-structured, distance when continuous)
  • Footer is ALWAYS 3 micro-stats: reference (target/ceiling/cap) + cadence + secondary temporal
  • State color migrates to the prescription metric — green/amber/red lives on whichever metric the workout cares about most
  • Delta chip is inline with the prescription metric, never floating elsewhere
  • Type: Bebas Neue italic for ALL labels and chips · Bebas Neue (non-italic) for numerals · tabular numerals enabled
WHY THIS WORKS WHILE RUNNING

Apple's research on the native Workout app: the runner's eye land sequence is fixed — top to bottom, left to right. By keeping label position constant (left edge of each row) and putting the BIG numeral immediately next to it, the runner doesn't re-scan; they just glance at the row they want. PACE → look at row 1. HR → row 2. Time left → row 3.

Bebas Neue's condensed letter width lets us fit a 64pt numeral plus its label plus an inline delta chip in a single row that's only ~75pt tall. At the same vertical real estate, SF Compact (Apple's watch font) at 64pt would consume more horizontal width and push the delta chip onto a second line.

Three big metrics + three micro-stats = six values total · same density as Apple's most detailed running view · but each big metric is unambiguously big.
