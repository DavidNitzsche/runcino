The arc of this session

You said "still getting Loading on every page load" + "push through all of it." Started by diagnosing + fixing the Loading flash (it was the cacheDate check + SSR-rendered loading text), then walked the queue in priority order.

FIX

Loading flash · root-caused + killed

Two compounding bugs: (1) readHubCachedSync was rejecting the cache when meta.cacheDate didn't match today's LA-calendar date, so the moment the day rolled over every revisit hit the loading state. (2) SSR was rendering the literal "Loading…" text, visible during the 100-200ms hydration window before useState's lazy initializer could populate from localStorage. Removed the date check; replaced "Loading…" text with transparent minHeight placeholders + aria-busy on every page. Bumped LS_KEY to v2 so v1 entries get GC'd. TTL now 24h since stale-while-revalidate handles freshness.

ENGINE

RPE consumption in readiness

The data was being captured but the engine wasn't reading it. Now state.rpe exposes recent (14d), avg7d, avgPrior7d, drift, recentHeavy. assessReadiness fires two new signals: "Perceived effort drifting up" when 7d-avg jumps ≥1.0 vs prior week (the canonical fitness-erosion canary), and "Recent session(s) felt heavy" when any of last 3 days hit RPE 8+. Drift ≥1.5 OR heavy-without-context shifts the verdict to YELLOW. Counts toward the doctrine cutback decision matrix.

TOOLING

Doctrine drift detector

scripts/check-doctrine-drift.ts. Walks app/+components/+lib/ looking for inline numeric literals matching values in coach/doctrine/. Filters trivial numbers + CSS-y lines. Currently flags ~838 candidates (mostly distance constants in non-doctrine code that happen to match doctrine numbers). Regex-based; AST analysis would tighten precision but the current version is signal-with-noise good enough for triage. Run via npm run check-doctrine-drift.
