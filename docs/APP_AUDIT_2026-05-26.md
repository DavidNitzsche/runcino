## 2. WHAT'S WORKING GREAT

- **Tool-use coach with truth contract.** Verified end-to-end: 7-9 tool calls per brief, voice anchored to today, threshold doctrine pulled from /Research/, no hallucinated runs.
- **Per-rep telemetry from watch → coach.** Watch ships `WatchCompletion` Data, lands in `coach_intents` + `strava_activities`, coach's `getWorkoutCompletion` tool reads it. Synthetic test confirmed coach can cite per-rep pace consistency + HR drift + cadence lock.
- **Background regen.** Mutations fire async warm. User mostly never waits.
- **Single source of truth across web, iPhone, watch, coach.** Same DB, same endpoints, same tables.
- **Daniels-style structured workouts** propagate cleanly from `plan_workouts` → web modal + iPhone card + watch payload — same shape everywhere.
- **/Research/ as the doctrine source.** Coach's `getDoctrine` tool reads canonical markdown, no general-knowledge filling-in on quality days.
- **Lock + atomic counter** is the right pattern for cross-agent ship arbitration.

## 3. WHAT'S BROKEN OR HAS GAPS

### iOS surface coverage gap
The iPhone has 5 tab views. The web has 12+ surfaces. Missing on iPhone:
- `/log` — no chronological run list
- `/runs/[id]` — no full run detail modal (taps on past-day tiles get the planned workout via WorkoutDetailModal, but no actual run review)
- `/races/[slug]` — no race detail view (only the season overview tab)
- `/tips` — no form-tips view
- `/settings` modal — no preferences UI on iPhone
- `/onboarding` — no onboarding flow
- `/learn/[slug]` — no learn-modal context

### Web `/onboarding`
File exists but appears stub. Onboarding state is pre-filled by David's data; new users would have no actual flow.

### Manual run entry — iPhone parity missing
Web has `/log` + `POST /api/run/manual` for backfilling. iPhone has no UI for this. If David runs without watch/Strava (treadmill, race forgotten to track), he'd need to use the web to log it.

### Shoe picker / rotation UI
Web has shoe CRUD on `/profile`. Neither web nor iPhone has a "which shoes did I wear on this run" picker. Shoe mileage isn't auto-incrementing on run completion. P15.2 planned, not built.

### Race retrospective form
Web has it. iPhone doesn't have a post-race form for capturing finish-time + how-it-felt + lessons.

### Legacy `/api/overview` shim
Still in place as 410. Once we're sure all legacy installs are gone, this can go.

### Cron secret stored in 2 places
Railway env + GitHub Actions secret. Rotation requires updating both. Not a bug; an operational tax.

### iPhone log of crashes
We almost flew blind on the launch crash because the Xcode Organizer path required David signing into Apple Account in Xcode. Worth automating a crash-pull path for future iterations.

### Build flakiness
Build 76 took 60+ minutes to surface in Apple's processing queue — almost shipped 78 thinking 76 was dropped. Apple's processing is just unreliable; not our bug, but worth knowing.

## 5. WHAT WE COULD ADD (proactive)

### David's idea — runner notes on completed runs

**Recommendation: BUILD it. P25 candidate.**

The case: telemetry tells the coach the runner ran 6.2mi at 145bpm. Telemetry can NOT tell the coach the watch was glitchy, the heat was worse than forecast, the legs felt like spaghetti, or the runner had a breakthrough mile they want to remember. That's a real signal gap. The runner already wants to talk about their run; the coach should be able to listen.

Implementation sketch:

- **Storage:** add `note: string` to `strava_activities.data` jsonb (no migration). Single text field per run, 0-500 chars.
- **UI:** text field on the run detail modal (web + iPhone). Auto-saves on blur. Placeholder: *"Anything the coach should know about this one? Quick notes welcome."*
- **API:** `PATCH /api/runs/[id]` accepts `{ note: string }` to update the field.
- **Coach access:** extend `getRuns` tool output to include `note` per run. Coach naturally absorbs them when discussing recent runs.
- **No cache bust on note add.** Per your instinct — notes are *context*, not events. They flow through on the next natural regen (day rollover, next run logged, etc.). The runner adding the note is a low-priority enrichment; the coach picking up the note later is the right time.
- **Surface in voice:** doctrine update in `coach/prompts/index.ts` post-run mode — *"If the runner left a note on the run (in getRuns output), absorb it into your framing. The note tells you what telemetry can't."*

Effort: ~80 LOC web + ~50 LOC iPhone + 10 LOC tool extension + 3 LOC doctrine. Half a day total.

## 6. RECOMMENDATIONS (priority order)

After build 77 lands and the iPhone launch crash is confirmed gone:

1. **Runner notes (David's idea)** — half-day, high signal, low risk. The right next move.
2. **Weather enrichment (P15.1)** — David already approved. Coach + run-detail both benefit.
3. **iPhone /log + /runs/[id] modal** — closes parity with web for the runner's most-frequent post-run flow ("what did I just do?"). Without this, the iPhone version still feels like a today-only app.
4. **Shoe picker + auto-mileage (P15.2 + P15.3)** — small but completes the gear-tracking loop.
5. **Plan adaptation triggers (B2)** — bigger swing. The "coach rewrites the next 7 days" feature David locked in last week. Needs design discussion before code.

What I'd PARK or DEPRIORITIZE:

- **iPhone race-detail surface** — race-day flow probably surfaces in a different way (watch-led on race day, iPhone glance only).
- **iPhone onboarding** — only matters if you're adding new beta testers.
- **HR ceiling alert on phone** — watch already handles it; phone duplication is low value.

## Bottom line

The architecture survives close inspection. Coach is honest, data flows from one source, mutations propagate, the daily cron warms the cache, and the watch ↔ iPhone ↔ web ↔ coach loop is real (verified end-to-end with synthetic post-run data).

The gaps are mostly iPhone-side surface coverage (log, run detail, settings, tips) and the v2 plan generator. None are blocking what the app already does.

David's runner-notes idea is the right next add — it solves a real signal gap with low effort and parks the rest of the backlog gracefully.
