Backend → Frontend.

Running log of every backend capability the system has, and whether it's actually surfaced in a frontend client. Maintained by the web agent, the iPhone agent, and the design agent as items move through the lifecycle below. Sorted by app surface. Plain English only.

Lifecycle · state machine for every row

Each row moves through five states. The state is encoded as a CAPS prefix on the Recommendation column · prefix followed by the existing body content (proposal, file ref, commit, etc).

  1. GAP · web/iOS agent flagged this · waiting for design judgment. Body: original recommendation flavor ("design tweak needed" / "can add inline · <proposal>" / "safe to leave · <why>").
  2. DESIGNING · design agent working. Body: short direction + expected file location.
  3. DESIGN-READY · spec ready · implementation queue. Body: designs/<file>.html · what's expected.
  4. IMPLEMENTING · web or iOS building. Body: branch name · target date.
  5. SHIPPED · merged to main. Body: commit <sha> · WEB / IOS / BOTH.

SAFE-TO-LEAVE · terminal · the implementation queue skips. Use for admin-only routes, internal derivations, capabilities that surface indirectly.

Who updates what · the design agent runs in Claude Design (canvas tool, no direct repo write) so the implementation agents own the lifecycle prefix updates. Design agent's memo references rows by surface section + capability name · implementation agents grep, find, update the prefix as work moves through.

How to use this doc · for the agents

You are building UI from a design. The design captures what the user is supposed to see. The backend captures what the system can do. When you encounter a backend capability — an API field, a table column, a cron output, a doctrine row, a coach trigger, a piece of computed state — that isn't represented anywhere in the design you're implementing, append a row to the matching surface section below.

The append format · one row per gap:

  • Capability: the thing the backend can do, named simply
  • What it does (plain English): one or two sentences a non-technical reader understands
  • Current coverage: NONE · PARTIAL · FULL
  • Recommendation: one of — "design tweak needed" (requires David to decide what the surface should look like) · "can add inline · <your proposal>" (you have a clean idea that fits the existing design language) · "safe to leave" (legitimately doesn't need a UI · explain why)
  • Source: the file:line or API route where you found it
  • Found by: WEB or IOS

Don't delete or modify other agents' entries. Only add. If the same thing is found twice, mention you saw it too with a note like "also found by IOS on 2026-05-31".

Do this as part of your normal work flow, not at the end. Discovery is highest-fidelity when you're elbow-deep in the code, not after you've already cargo-culted past it.
