The North Star

A runner who opens the app cold sees, within 5 seconds, exactly what to do today, why, and how it fits their goal. Every number traces back to a source. The coach has memory across weeks, asks how they feel, names patterns they don't see, and pushes back when goals are unrealistic.

The app is coaching, not just dashboarding. We're roughly 60% of the way there. The remaining 40% is what this document maps. The unifying principle: one source of truth, many ways to use it — the canonical RunnerHub backs every surface, every page is a projection of it.

Critical findings worth your attention

7 misleading-class issues

Long-run cap math disagrees between dashboard and engine

TrainingPulseTile (× 0.6 / 0.5 / 0.7 for TAPER / POST-RACE / REBUILD) vs engine longRunTarget (× 0.65 / 0.4 / 0.6). Runner sees one cap on the dashboard and a different long-run distance on the prescription. Direct trust erosion.

Server-side coach state has no runner identity

Age, sex, HRmax, RHR live in localStorage. The brief LLM never knows who the runner is — can't say "exceptional VDOT for a 60-year-old woman." Engine's "yesterday hard" gate is hardcoded 152 BPM regardless of HRmax. This is the #1 unlock — closing it makes ~5 doctrine pathways consumable at once.

/workout/[date] is 100% static placeholder

Linked from the dashboard week-strip, the 30-day strip, and the training-page next-up list. Three high-traffic doors all open into a frozen design canvas.

FuelingTile uses planner default carb/hr, not doctrine

Should pull from RACE_CARB_TARGETS_G_PER_HR like the new HydrationTile does. Same parallel-source-of-truth problem.

CoachTodayCard.hrZone is a static 1–5

Not derived from runner HRmax. A 60yo reading "HR Z4" gets misleading prescription guidance. Closes when bug #2 closes.

Training page is essentially redundant with dashboard

Two daily voice paragraphs, two week-shapes, two prescription views. Not technically a bug — a product issue. Section 4 below proposes the split.

Engine has parallel post-race ladder duplicating doctrine

Hand-coded ladder in coach-engine.ts:271-321 when recovery_protocols.ts.POST_RACE_BY_DISTANCE + REVERSE_TAPER_PROTOCOL already exist. Same shape, two sources, easy to drift.

The dashboard, rethought

From flat 14-tile stack to mode-aware hierarchy

Today: a flat stack of 14 tiles, every weight equal. A runner on race-day morning sees the same hierarchy as one on a recovery Tuesday. The thing that matters most for THIS runner THIS minute is buried mid-page.

Tomorrow: 9 distinct runner modes, each with its own hero tile elevated above the fold. The metric strip and explorers stay; the hero changes by context. Same component library, different ordering per useRunnerMode().

6 new pages worth building

From Section 8g

Strategic

/season
Bird's-eye view of the runner's race calendar across the year — race blocks, recovery windows, base periods, fitness projection over months.
Top 3: Race timeline · Phase blocks · Conflict detection (overlapping recoveries, peak collisions).

Calibration

/calibration
Every dial in one place — pace zones, HR zones, fueling rate, hydration rate, sweat rate, stride cadence target. The runner's full coaching profile on one canvas.
Top 3: All zones (pace + HR) · Fueling targets · Sweat / hydration calibration.

Today (focused)

/today
When the dashboard is too much. Just today's workout, the brief, the warm-up sequence, the cool-down. Pre-run focus mode.
Top 3: Today's workout · Voice brief · Warm-up checklist with timer.

Library

/library
All workout types in the doctrine, browsable. When did you last do an MP block? What does Norwegian double-threshold actually look like? Each opens to definition + structure + when-to-use + last-attempt.
Top 3: Workout type list · "Last done X ago" · Cited research per type.

Research

/research
Browse the doctrine itself. Every research doc viewable in-app, with cross-links between them. The "why" behind every coaching decision exposed for runners who want to dig.
Top 3: 24-doc index · In-app viewer · Cross-references and citations as links.

Patterns

/patterns
Recurring patterns the runner doesn't see. When they tend to skip, when they push too hard, what time of day they run best, which Tuesdays go to threshold vs easy.
Top 3: Skip / overtrain patterns · Day-of-week tendencies · Goal-vs-actual deltas.

Decision time

Path forward

Recommended path

Architecture first (weeks 1–3) · then redesign (weeks 4+)

The redesigns assume the hub-as-backbone is in place. Building the dashboard mode-dispatcher before wiring the runner profile to the server means re-doing it after week 1 anyway. Front-load the unlocks. Back-load the polish.

The single highest-impact next move: recommendation #1 — wire runner profile to server-side CoachState. Closing it unlocks #5 (recovery_protocols), #10 (HRmax gate), and the brief LLM finally seeing demographics. The 4-week roadmap above is structured to do exactly that, then build downward.
