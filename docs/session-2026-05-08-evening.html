SHIPPED · ARCHITECTURE

RunnerHub migration · 6 phases

Every page in the app now projects from useHub() — dashboard, training, races, race-detail. One /api/hub endpoint fans out coach + races + profile + RPE in parallel. Mutations bump one cache; every subscribed page re-renders. The phone vs desktop disconnect is gone.

SHIPPED · DOCTRINE

Long-run cap math unified

The engine and the dashboard each had inline copies of per-phase cap multipliers (TAPER × 0.65, POST_RACE × 0.4, etc). Extracted to lib/long-run-cap.ts as the single source — both surfaces now call longRunTargetMi(phase, peakLast28Mi). Drift is impossible.

SHIPPED · DOCTRINE

Post-race recovery from doctrine

Engine's recovery ladder used to be hand-coded (3/7/14/26 days). Now reads POST_RACE_BY_DISTANCE from Research/00b across 8 distance bands (5K through 100mi). Stage gates derive from zeroOrVeryLightDays, returnToLongRunsDay, totalRecoveryDaysNoQuality. Marathon recovery extended 26→28 days (more conservative).

SHIPPED · DOCTRINE

Sweat-rate + EAH risk surfaced

HydrationTile on race-detail now also shows: full Research/19 sweat-rate calibration protocol (formula, steps, example, 4 typical-range categories) + 9 EAH risk factors as a warn-tinted alert. Generic ml/hr bands aren't enough; sweat rates vary 10× across runners.

SHIPPED · INPUT

Workout RPE logger

New workout_rpe Postgres table + /api/rpe endpoint + RpeInput component (Borg 1-10 scale, instant save). Mounted on the dashboard and the new /workout/[date] page. Hub now carries recentRpe so the dashboard shows a 7-day mini-history strip.

Doctrine source: Research/00b §INCOMPLETE_RECOVERY_QUALITATIVE_SIGNALS. Engine consumption of the signal is the next step — this commit captures the data so the signal exists.

SHIPPED · PAGE

  • Phone and desktop see the same races + profile + RPE. All on Postgres now; the localStorage masking is gone.
