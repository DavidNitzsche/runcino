Why your phone was showing different info

Diagnosed + fixed

Both root causes fixed

Loading text lied + runner profile lived only on the device you set it on

(1) The loading text said "Reading saved plans from localStorage…" — leftover from when storage WAS local. Race plans have actually been in Postgres for a while; the message was a UI lie. Now says "Loading races…".

(2) Runner profile (birth year, sex, HRmax, RHR) was browser-local only. Set on desktop → desktop showed age-graded VDOT and HR zones. Phone had nothing → phone showed raw VDOT only. That was the real disconnect.

Now the profile lives in a runner_profile Postgres table:

  • Cross-device synced — phone, desktop, future iOS app all read the same profile
  • Auto-migrates legacy localStorage profile on first load (preserved as backup)
  • Server-side coach engine sees it now — was the audit's #1 priority. The brief LLM gets age + sex + HRmax in its prompt. The "yesterday hard" gate uses the runner's actual HRmax (was hardcoded 152 BPM)
  • Single-row table for now (no auth yet, one runner per deploy). Migration path to per-user when auth lands.

Option C — coach today cache

Built fully · 3 trigger paths

Every page nav used to fire /api/coach/today, which re-ran gatherCoachState, the engine, AND the LLM brief call (~2-5s, ~$0.005-0.015 per call). With 5+ tiles each fetching independently before today's earlier dedupe, that was 5x cost. Now:

┌─ Dashboard load ──────────────┐ │ GET /api/coach/today │ │ ├─ readCache(date, activityId) │ │ ├─ HIT → return cached payload (~10ms) │ │ └─ MISS → fall through │ └─ computeFullPayload() │ ├─ gatherCoachState (Postgres + Strava cache) │ ├─ coachDaily + readiness + workout │ ├─ briefDailyTraining LLM call (~2-5s) │ └─ writeCache + return └───────────────────────────────┘ Cache key: (cache_date, latest_activity_id) Auto-invalidates when: - Date rolls over (new "today") - Latest Strava activity ID changes (new run) Three trigger paths land the cache HOT before you visit: ① Strava webhook POST /api/strava/webhook → activity create → regenerate ② Midnight cron GET /api/cron/coach-refresh → bearer-auth daily refresh ③ Lazy fallback GET /api/coach/today → recompute + write on miss

Setup steps when you have a moment

Cache is live; webhook + cron need env vars

~10 minutes of setup

Webhook + cron need env vars + a curl call

Without these, the cache still works — just lazily (first visit after a run pays the 2-5s LLM cost once instead of being instant). With them, every visit is instant.

  • 1. Set STRAVA_WEBHOOK_VERIFY_TOKEN to any random string in Railway env vars.
  • 2. Set CRON_SECRET to a different long random string in Railway env vars.
  • 3. Register the webhook with Strava (one-time curl): 
    # Run this once to subscribe Runcino's webhook with Strava.
curl -X POST https://www.strava.com/api/v3/push_subscriptions \
  -F client_id=$STRAVA_CLIENT_ID \
  -F client_secret=$STRAVA_CLIENT_SECRET \
  -F callback_url=https://runcino-production.up.railway.app/api/strava/webhook \
  -F verify_token=$STRAVA_WEBHOOK_VERIFY_TOKEN
  • 4. Add a Railway cron job hitting https://runcino-production.up.railway.app/api/cron/coach-refresh with header Authorization: Bearer $CRON_SECRET. Schedule: 1 7,8 * * * (covers both PT and PDT just-after-midnight).
