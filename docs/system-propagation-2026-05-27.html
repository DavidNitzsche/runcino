TODAY's pattern is the unit: poster + sibling + mini-tile dashboard. Every other page is the same three components asking a different question. Plan asks where am I in the cycle? · Run-detail asks did I hit it? · Races asks am I ready? · Health asks what does the body say?

the principles · how to extend the system anywhere

three rules
for every page.

same building blocks · different content · zero design debt
01
one question

Every page asks one question.

TODAY asks "what am I doing?" · PLAN asks "where am I in the cycle?" · RUN DETAIL asks "did I hit it?" · RACES asks "am I ready?" · HEALTH asks "what does the body say?"

If you can't write the page's question in 5 words, the page isn't ready to design yet. The hero verb is the answer to the question.

02
three blocks

Poster · sibling · body.

Poster (gradient card, hero verb, stat trio) answers the question. Sibling (mini-tile dashboard or prose+tiles) gives supporting context. Body below (two-column) shows depth on demand.

Each page composes the same three structural components. Poster · Sibling · BodyGrid are SwiftUI/React components reused everywhere — only the content differs.

03
gradient = identity

Color tells you where you are.

Every page gets a gradient family — TODAY's gradient is its state (easy/quality/long/etc), PLAN's gradient is purple-blue (the phase cycle), RACES is orange-red, HEALTH is cyan, LOG is muted slate.

You should recognize what page you're on from the gradient before reading the title. Color is wayfinding.

The answer to your question

Two layers. The 13-segment bar shows the shape of the workout (warmup · 6 reps with rests between · cooldown), proportional to time spent. Below it: a per-rep grid where every rep is its own tile.

Pre-run: tiles show targets in amber. Mid-run (LIVE state): the current rep highlights, others greyed. Post-run: tiles fill with actuals + colors (green = hit, amber = soft, red = missed).

Below the grid: rest interval · HR target · pace context (what race the pace correlates to). Everything a runner needs without leaving the page.

the common shell

one card · one ratio · five charts inside.

Every variant lives in the same workout-breakdown card: header row (workout name + total dist/time) · the chart proportional to the structure · per-segment data row · optional footer (fuel · HR · pace context). Same chrome, different chart.

The chart type is picked off workout.type: easy → 3-segment bar · intervals → 13-segment bar + rep grid · tempo → 3-phase proportional bar · long → mile-marker rail with fuel cues · hills → elevation profile · progression → ascending pace ladder.

Pre-run the card shows targets. During the run the active segment lights up. Post-run the chart fills with actuals + result colors (green=hit, amber=soft, red=missed). Same component, three states.

implementation · what this saves us

one system · six pages · six workout variants.

Component library is small. Poster · SiblingCard (with MiniTileGrid child) · WeekStrip · BodyGrid · WorkoutBreakdown (with chart-variant child) · StatTrio · BCard. Eight shared components carry every page in the app. Reuse instead of rebuild.

The resolver becomes the page-level entry. Each page has a thin resolveX(state) function that returns the right gradient, verb, sibling content, body content. TODAY's resolver returns one of 11 day-states. PLAN's resolver returns one of 4 phase states. RUN-DETAIL returns one of 7 result-keyed verb states. RACES returns one of 5 race-proximity states. Same pattern · different content.

Workout-breakdown is its own little library. 6 chart components — one per workout type — sit inside the breakdown card. The card picks the chart based on workout.type. Pre-run shows targets · live shows current segment · post-run shows actuals with result colors. Once these 6 charts are built, every workout the plan asks for can be rendered honestly — no fallback to "see the workout" prose.

Brand consistency is automatic. Gradients are CSS variables. Verbs come from a lookup table. Mini-tiles use the same component. Week strip is the same across TODAY and PLAN. The system is the design. New pages inherit it for free.

4 sprints to ship the system app-wide:

  • Sprint 01 · TODAY (1 week) — Resolver + components + 4 base states (EASY/QUALITY/LONG/REST). 70% of daily home views.
  • Sprint 02 · TODAY edges + RUN DETAIL (1 week) — DONE/MISSED/NIGGLE/SICK states, then RUN DETAIL page (it reuses ~80% of components).
  • Sprint 03 · PLAN + RACES (1 week) — Phase resolver + 8-week grid · race detail with trajectory + race-day ops sibling.
  • Sprint 04 · HEALTH + LOG + PROFILE + workout-breakdown library (1 week) — Trends charts, history feed, identity surface · all 6 workout-breakdown chart variants.
