The bad news, summarized

I found 11 gaps beyond the HK workout reader. Three are critical — silently broken or about to break. Four are stubs (fake/hardcoded data masquerading as live). Four are declared-but-orphaned (entitlements / endpoints / fields with no actual code path).

The most urgent: your daily HRV / sleep / RHR sync depends on the LEGACY iPhone app still being installed. The v2 app has zero HK sample-reader code. If you uninstall legacy, all biometric data stops flowing.

Total scope to close all 11: roughly 1.5–2 days. Worth doing in a focused pass.

Critical gaps · silently broken or about to break

CRITICAL · P27.1

iPhone v2 has zero HealthKit sample reader (HRV / sleep / RHR / VO2 / cadence)

The new HealthKitImporter.swift I just built only reads HKWorkout. It does NOT read any biometric samples. All your daily HRV, sleep hours, resting HR, VO2max, wrist temp, cadence baseline — that's flowing exclusively from the legacy iPhone app's HealthKitManager.swift, which posts to /api/ingest/health.

If you uninstall the legacy app (or it gets de-authorized for HealthKit), every readiness calculation goes blind: no sleep, no HRV trend, no RHR baseline. Coach starts saying "I can't see your readiness data." Probably tomorrow.

Evidence: grep -rn ingestHealthSamples native-v2/ → zero matches. grep -rn HKQuantityType native-v2/ → only my new file, and only the workout-specific types.

Fix: Port the sample-collection path from legacy/native/Faff/Faff/HealthKitManager.swift (the collectSamples function — lines 201–360 — already pulls every type the backend accepts). Wire to /api/ingest/health. ~150 LOC.

Severity: data quality silently degrades; coach blindness on readiness within a day or two.

CRITICAL · P27.2

iPhone calls /api/readiness and /api/watch/readiness — neither endpoint exists

The iPhone code references both endpoints. The server has no route handler for either. Requests get a 404 HTML page. Failures are swallowed silently.

Visible consequence: the readiness ring on iPhone TODAY is ReadinessChip(value: 88) — a literal 88 hardcoded in Swift. Comment in the code: "Readiness ring — wired with real score when /api/readiness ships."

Evidence: ls web-v2/app/api/readiness → no such file. grep -n readiness web-v2/app/api/watch/today/route.ts → zero matches (the watch-today endpoint doesn't return readiness fields either, but the WatchWorkout model expects them and decodes nil).

Fix: Build GET /api/readiness server endpoint that computes the score from health_samples + check_ins (the logic already lives in web-v2/lib/coach/state-loader.ts as getReadiness — wrap it in a route). Wire iPhone ReadinessChip to the real value. ~50 LOC server + 30 LOC iPhone.

Severity: highest-visibility iPhone surface shows a fake number. Coach's "readiness=88" you've been seeing on the phone has never been your real readiness.

CRITICAL · P27.3

Duplicate-run merge infrastructure is read-only in v2

Every state-loader in web-v2/lib/coach/ filters out runs flagged data.mergedIntoId. That's the de-dupe guard. But nothing in web-v2 ever WRITES that flag. The write code lives only in legacy/web/lib/run-merge-overrides.ts and legacy/web/app/api/runs/dedupe-backfill.

So today, when a run lands via three paths (Strava webhook + HKWorkout + Faff watch-app), each creates a separate row. None get merged. The coach sees three runs instead of one. The log shows three.

This explains your earlier "coach hallucination: watch synced the threshold workout from yesterday" — duplicate ingests with no merge.

Evidence: grep -rn 'mergedIntoId.*=\|jsonb_set.*mergedIntoId' web-v2 → zero. The flag is consumed but never produced.

Fix: Port the merge logic from legacy/web/lib/run-merge-overrides.ts to web-v2/lib/runs/merge.ts. Call it from every ingest endpoint (/api/ingest/workout, /api/run/manual, /api/watch/workouts/complete, Strava webhook). Match on (user, start_time ± 60s, distance ± 10%). ~120 LOC.

Severity: every dual-source run is double-counted today. Affects log totals + coach weekly volume calculations + plan adaptation.

Stubs masquerading as features

STUB · P27.4

Web /onboarding partly persists, partly silently drops

web-v2/app/onboarding/page.tsx is 34 lines — just the wrapper. The flow is in web-v2/components/onboarding/OnboardingFlow.tsx (247 lines). It saves the race via POST /api/race. But a code comment explicitly says:

"for now we just don't persist them — no fake CONNECTED bits" — referring to Strava + Apple Health connection statuses.

If a new user clicks "Connect Strava" or "Connect Apple Health" in onboarding, nothing happens server-side. They proceed thinking it worked.

Fix: Wire real connection-state persistence to profile.strava_connected_at + profile.health_connected_at columns. Either build real OAuth or be honest about the placeholder. ~60 LOC.

STUB · P27.5

Watch face decodes readinessScore / readinessLabel — server never sends them

The watch's WatchWorkout model has readinessScore: Int? and readinessLabel: String?. The server endpoint that ships this model (/api/watch/today) doesn't populate either field. Watch decodes them as nil. Falls through to the workout fixture data (literal "Primed" / "82" in the legacy app).

Fix: Same fix as P27.2 — once /api/readiness exists, pipe it into /api/watch/today output. ~10 LOC after readiness endpoint exists.

STUB · P27.6

Watch fixture data has hardcoded "Primed · 82" placeholders

legacy/native/Faff/FaffWatch Watch App/WatchWorkoutModels.swift line 366 has a fixture: readinessScore: 82, readinessLabel: "Primed". This shows up if the watch fails to decode a real payload (e.g. server returns no readiness).

Fix: Either remove the fixture default or replace with a "no data" state. The watch shouldn't lie about readiness. ~5 LOC.

STUB · P27.7

shoes.mileage_mi never auto-increments — even though shoe CRUD works

The strava_activities table has a shoe_id column. /api/shoe supports POST/PATCH/DELETE. Web profile page renders the shoe rotation chart.

But nothing ever writes shoe_id on a run. No UI to pick a shoe. No auto-assignment. shoes.mileage_mi stays at whatever you typed in when you created the shoe.

Fix: Shoe picker on the run detail modal + auto-bump on save. ~120 LOC. Already P15.2 in the queue.

Declared-but-orphaned (entitlements, fields, capabilities without code)

ORPHAN · P27.8

NSHealthUpdateUsageDescription declares write capability the iPhone never uses

iPhone Info.plist declares "Faff writes completed workouts to Apple Health so your history stays in one place." Native-v2 has zero HK write code — that's the watch's job. App Store reviewers might flag this as misleading.

Fix: Either implement HK workout write-back from iPhone (probably no — watch already does it), or remove the usage string. ~5 LOC.

ORPHAN · P27.9

temp_f field in run data — never populated

The strava_activities.data.tempF field is in the schema and read by the coach via run-state.ts. Nothing writes it. All runs have null temp. Coach can never speak to heat.

Fix: Open-Meteo enrichment — P15.1, already approved. ~80 LOC.

ORPHAN · P27.10

WatchSync authToken TODO — multi-user is a no-op

Comment in WatchSync.swift: "authToken — TODO wire when v2 auth lands (P6). Watch tolerates absence."

The watch + iPhone identify the user by hardcoded DEFAULT_USER_ID on the server (35 occurrences). No bearer tokens, no sessions, no JWT. The system is single-user. Adding a second beta tester today would either mix their data into yours, or require redeploying with their UUID hardcoded.

Not necessarily a bug — beta is intentionally single-user. But it's a hard ceiling. Bear in mind before any TestFlight expansion.

Fix when needed: Real auth (Sign in with Apple is cheapest) + JWT + multi-tenant queries. ~400-600 LOC across both clients + server. Defer until needed.

ORPHAN · P27.11

Strava OAuth wired by env vars — no per-user connection flow

Strava is connected via STRAVA_CLIENT_ID + STRAVA_REFRESH_TOKEN in your .env.local. There's no web OAuth flow, no "Connect Strava" button that actually does anything beyond UI display. The runner-tagged tokens are yours specifically. Onboarding's "Connect Strava" is the same placeholder mentioned above.

Same family of finding as P27.10 — single-user shortcut.

Fix when needed: Real Strava OAuth flow (auth URL → callback → store per-user refresh token). ~150 LOC. Defer until multi-user.

The pattern — why this happens

How "looks built, isn't" got past us

Three causes, in order of frequency:

  1. The audit checked surfaces, not code paths. The morning audit confirmed "iPhone TODAY · live" because the screen renders. But the screen rendering a "88" readiness ring isn't the same as readiness actually working. Surface-level audits miss placeholder content.
  2. Legacy was the source of truth, never officially deprecated. When we cut to native-v2, half the iPhone's functionality (HK sample sync, run de-dupe) silently kept flowing from the legacy app because David still had it installed. We never tested with the legacy app uninstalled, so we never noticed.
  3. Schema fields outlived their consumers. mergedIntoId, shoe_id, tempF are all schema columns that ship with read code (state-loader, coach tools) before write code lands. Easy to think the feature exists because there's data structure for it.

Going forward: when an audit says "X is live", the check should be: "trace the data path end-to-end, on the iPhone built today, with legacy uninstalled." Surface-and-grep isn't enough.
