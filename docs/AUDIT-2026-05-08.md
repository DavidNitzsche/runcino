# Runcino Codebase Audit — 2026-05-08

A synthetic review of the marathon-coaching app's three-layer architecture (research doctrine → engine/state → UI), focused on what's actually wired, what's documented but inert, and where the next round of value is hiding.

**Scope:** `web/` codebase at branch `claude/objective-black-8f3e69`, Research doctrine at `/Volumes/WP/06 Claude Code/Runcino/Research/`.

**Method:** static read of every page, every doctrine constant export, every consumer import path. References are file:line where it matters.

---

## 1. Research doc coverage matrix

The Research/ folder has **24 numbered .md files** plus 5 meta files (INDEX, GLOSSARY, SOURCES, RESEARCH_TASKS, REVIEW_NOTES). The `web/coach/doctrine/` folder has **34 .ts files**. The mapping is roughly 1:1 by topic but not by file — some research docs were split into multiple doctrine files (e.g. workouts.ts pulls from 04 + 05 + parts of 03), and several large doctrine files exist as scaffolding that nothing reads yet.

A doctrine file is **WIRED** when at least one of its constants is imported and consumed by `web/lib/*` engine code, `web/coach/coach.ts`, `web/coach/explanations.ts`, or a `web/app/api/*/route.ts`. The actual consumer set is small — `coach-principles.ts`, `vdot.ts`, `coach-plan.ts`, and `weather-slowdown.ts` are essentially the only code paths that read doctrine.

## 3. State + data flow

`gatherCoachState()` lives in `web/lib/coach-state.ts:154-313`. It is invoked from `/api/coach/today` and `/api/brief` exclusively (server-side).

### Reads
- **Postgres**: `listRacesDB()` from `lib/race-store.ts` (saved races + actualResult)
- **Strava cache**: `getCachedActivities()` from `lib/strava-cache.ts` (NormalizedActivity[])
- (HealthKit is in the type signature but always null — `flags.healthKitAvailable: false` line 309)

| `flags.rebuildAfterBreak` | last7Mi ≤ 30% of last28Mi/4 | engine, readiness, voice, brief context |
| `flags.healthKitAvailable` | always false | nothing |
| `recoveryWindowEndsISO` | latest race-recovery end among recent races | engine `decidePhase` (POST_RACE) |

### Computed but never read
The following CoachState fields are computed but no consumer accesses them:
- `races.nextAny` (always equals or supersets `nextA`)
- `races.raceCount30d` (only used in a single rationale string — could be inlined)
- `volume.last7Days` (UI computes its own week-day breakdown from Strava)
- `intensity.easyMi14d`, `intensity.hardMi14d` (only `easyShare14d` is read)
- `recovery.consecutiveRunDays`
- `recovery.hrv7dAvgMs`, `recovery.rhrBpm`, `recovery.sleep7dAvgHrs`, `recovery.strengthDaysThisWeek` (M2 placeholders)
- `flags.healthKitAvailable`

This is normal for an in-progress build but worth flagging — the HealthKit fields advertise capability the engine doesn't yet use even when available.

### Critical gap: CoachState has no
- runner age / sex / weight (despite age.ts and sex.ts doctrine)
- HRmax / LTHR (engine assumes 152 bpm = 80% × 190 default)
- injury flag
- travel state (timezone deltas, jet-lag stage)
- training preferences (long run day from /profile is local-only, never persisted)
- explicit phase override (user can't say "I'm in build phase")
- carb tolerance / GI calibration delta (the type definition mentions it for fueling but it's never persisted)

## 4. Coach prompts inventory

`web/coach/coach.ts` defines the `Coach` interface (lines 240-275) with **7 methods**. Implementation in `CoachImpl` (line 282).

### Method status

| Method | Stage | Implemented? | Brain |
|---|---|---|---|
| `paceStrategy` | 1 | **Stub** (throws) | — |
| `taperDepth` | 1 | **Stub** | — |
| `fuelingFor` | 1 | **Stub** | — |
| `prescribeWorkout` | 3 | Wired | Deterministic |
| `assessReadiness` | 3 | Wired | Deterministic |
| `briefRaceMorning` | 2 | Wired | LLM with deterministic fallback |
| `retrospect` | 4 | **Stub** | — |
| `adjustForReality` | 5 | **Stub** | — |

Three methods are wired; four are stubs that throw `Coach.X() lands in Stage N`. `paceStrategy` / `fuelingFor` are particularly notable as stubs — the actual race-pacing and fueling logic still lives in `lib/pacing.ts` and `lib/fueling.ts` / `lib/fueling-claude.ts`, called directly by `lib/build-plan` and the race-rebuild API. Coach is supposed to be the single entry point but it doesn't yet own these surfaces.

### `prescribeWorkout(input: PrescribeWorkoutInput)` (coach.ts:298-333)

- **Input**: `{ today: ISO, state: CoachState }`
- **Body**: wraps `coachDaily(state)` from `lib/coach-engine.ts`. Extracts run, attaches `composePhaseLabel` for the phase chip, calls `composeVoiceLead` from `coach/explanations.ts` to build the multi-sentence body paragraph, picks `citationsForWorkoutType` from `coach/citations.ts`.
- **No LLM**. Pure deterministic.
- **Output**: `WorkoutPrescription` (label, distance, paceTarget, hrZone, phaseLabel, voiceLead, isQuality, isLong, full coachToday).
- **Consumers**: `/api/coach/today` → `app/page.tsx CoachTodayCard` and `app/training/page.tsx DailyBriefing`.

### `assessReadiness(input: AssessReadinessInput)` (coach.ts:340-398)

- **Input**: `{ today: ISO, state: CoachState }`
- **Body**: pure deterministic. `acwr(state)` ratio + heavy-block / post-race overrides + easyShare / missedRuns thresholds, decision tree returning `green | yellow | red`. `citationsForReadiness(level)` adds 2-3 citations.
- **Output**: `ReadinessAssessment` (level, message, acwr, easyShare).
- **Consumers**: `/api/coach/today`. The training page reads it from the JSON response but it's not visibly rendered in the current `DailyBriefing` layout (the readiness sentence used to render separately; the new design folds context into the voiceLead).

### `briefRaceMorning(input: RaceMorningBriefInput)` (coach.ts:406-644)

- **Input**: `{ today, raceName, raceDate, goalDisplay, weather?, courseSummary, elevationFt?, goalPaceSPerMi?, raceDistanceMi?, abilityTier?, altitudeAcclimatized?, daysToRace?, trainingContext? }`
- **Body**:
  1. Computes `weatherSlowdown` deterministically via `lib/weather-slowdown.ts` (which reads weather.ts doctrine — `MAUGHAN_HEAT_SLOWDOWN`, `DEWPOINT_PACE_ADJUSTMENT`, `ALTITUDE_RACE_LOSS`, `WIND_PER_MILE_COST`, plus bail-flag logic from `QUALITY_SESSION_BAIL_TRIGGERS` and `HARD_CANCEL_TRIGGERS`).
  2. Picks horizon from `daysToRace`: morning (≤0), race_week (≤7), approach (≤21), course (>21). Each gets distinct prompt instructions (lines 552-576).
  3. Builds a training-state read (lines 583-621): VDOT vs goal time delta with sentence-level verdict, recent volume picture, longest run, easy share, heavy-block / rebuild flags.
  4. **LLM brain when available** (`callCoachLLM` at line 638): user prompt is the assembled context; system prompt is in `coach/voice.md` (loaded via `coach/llm.ts`). Coaching-research.md is cached as the first system message (per llm.ts) so the model has the full doctrine in context.
  5. **Deterministic fallback** (lines 425-510): per-horizon templated copy with weather clause (5 tiers based on slowdown %), training-line one-liner, citations referencing Research/06 §10 + Research/08 §3.5.
- **Output**: `CoachDecision<string>` — the brief paragraph plus rationale + citations.
- **Consumer**: `/api/brief` → `useAdaptiveBrief()` hook in race detail page → `CoachBriefBlock` rendered inside the PosterCard description column.

### Stubbed methods

`paceStrategy`, `taperDepth`, `fuelingFor`, `retrospect`, `adjustForReality` all throw on call. Active surrogates:
- Pacing: `lib/pacing.ts` (Minetti grade-adjusted-pace direct call)
- Taper: implicit in `lib/coach-principles.ts:weeklyVolumeMultiplier` and the engine's TAPER phase logic
- Fueling: `lib/fueling-claude.ts` (Claude SDK call) with `lib/fueling.ts` rule-based fallback — both bypass the Coach
- Retrospect: `/api/retrospective` calls `lib/retrospective.ts` + Anthropic SDK directly; uses its own RETROSPECTIVE_SYSTEM_PROMPT not voice.md
- adjustForReality: not exposed anywhere

This is the largest architectural debt in the file: the Coach interface promises to be the single judgment surface but four of its seven methods are still bypassed.

## 5. Gap analysis — high-value info we HAVE but don't show

Cross-referencing sections 1-4, here are concrete cases where Research is documented, doctrine is structured, but no UI surface exposes it.

### 5.1 HR zones (Research/03 + `hr_zones.ts`)
- Doctrine has 23 exports including `HRMAX_ZONES_5`, `LTHR_ZONES_5`, `KARVONEN_ZONES`, `HRV_INTERPRETATION_PATTERNS`, drift tolerance, decoupling thresholds.
- App uses **a single hardcoded 152 bpm** (`coach-principles.ts:239`) for the "yesterday was hard" gate.
- The dashboard's HR-zone display on the VDOT tile is in pace zones, not HR zones — there's no actual HR zone surface anywhere in the app.
- The /runs/[id] page shows avg/max HR but never shades them by zone, never compares to LTHR.

### 5.2 Hydration / fueling beyond gels (Research/19 + `hydration.ts`)
- 12 hydration constants documented; zero consumed.
- The race-detail FuelingTile shows gels with their carb count — but never sodium target, never water volume, never EAH risk warnings, never sweat-rate intake protocol.
- A runner doing Big Sur in July with no sodium plan gets nothing.

### 5.3 Pre-race carb load + meal (Research/18 + `fueling.ts`)
- `CARB_LOAD_24_48HR.value = { gPerKgPerDayLow: 8, gPerKgPerDayHigh: 10, glycogenUpliftPctLow: 20, glycogenUpliftPctHigh: 40 }` — defined at `fueling.ts:48-58`.
- `PRE_RACE_MEAL.value = { gPerKgLow: 1, gPerKgHigh: 4, hoursBeforeLow: 1, hoursBeforeHigh: 4 }` — line 61-74.
- Neither shows up anywhere. The race-week brief variant in `coach.ts:559-562` mentions "carbs" generically but doesn't pull the per-kg numbers.

### 5.4 Recovery protocols catalog (Research/00b + `recovery_protocols.ts`)
- 22 constants. The most consequential: `POST_RACE_BY_DISTANCE`, `MARATHON_BIOMARKER_TIMELINE` (CK/myoglobin returns to baseline ~14 days), `REVERSE_TAPER_PROTOCOL`, `MULTI_RACE_CADENCE`, `CARBON_PLATE_RECOVERY_EFFECTS`, `INCOMPLETE_RECOVERY_DECISION_MATRIX`.
- The engine has its own ad-hoc post-race ladder in `coach-engine.ts:228-278` — works but disagrees with doctrine on day-by-day specifics.
- No reverse-taper visible to the user. A runner who finished a marathon last weekend sees "post-race recovery" as a phase chip but no day-by-day reverse-taper plan.

### 5.5 Training intensity distribution recommendation (Research/00a §3.1 + `intensity.ts:69-85`)
- `PHASE_DISTRIBUTION_RECOMMENDATION` says: pyramidal in base/build, polarized in peak. Defined and never consumed.
- The engine's `intensityTarget()` in `coach-principles.ts:82-92` chooses easyShareMin/Target/qualityDaysPerWeek by phase but doesn't reference the distribution model name. A user asking "am I doing pyramidal training" cannot tell from the UI.

### 5.6 Strength periodization (Research/07 + `strength.ts`)
- `STRENGTH_PERIODIZATION` published; engine in `coach-principles.ts:201-222` re-encodes it by hand and lands at the same rules — but reads from `coach-principles` not from doctrine. Two parallel sources.
- `HEAVY_RESISTANCE.primaryAdaptation: 'running_economy'` is a key insight (strength training raises economy, not VO2). Never surfaced. The strength tile on the dashboard says "Heavy resistance — Build maximum-force capacity" without explaining this is for running economy.

### 5.7 Cross-training substitution (Research/09 + `cross_training.ts`)
- `XT_DECISION_RULES`, `XT_CARRYOVER_MATRIX` (specificity ranking AlterG > DWR > Elliptical > Stair > Cycling > Rowing > Swimming).
- A runner who emails "knee hurts, what should I do today?" — no UI to log injury, no XT prescription, no equivalent-stimulus calculator.

### 5.8 Mental training (Research/20 + `mental.ts`)
- `GOAL_SETTING_FRAMEWORKS` includes A/B/C goals (line 22-26): A = stretch <30% probability, B = realistic 50-70%, C = insurance 95%+. The race entry form has only `goalDisplay` as a single string — no A/B/C.
- `PETTLEP_VISUALIZATION` is a 7-component visualization protocol with race-specific timing. Race-morning brief never references it.
- Self-talk catalog and pre-race anxiety protocols all unused.

### 5.9 Travel + jet lag (Research/12 + `travel.ts`)
- `TRAVEL_ARRIVAL` recommends arrival days by time-zone-class.
- `EAST_WEST_ASYMMETRY` says eastward is harder, prescribes 30-60 min/day bedtime advance for 3-5 days.
- No travel field anywhere. A runner flying from LA to a Boston race gets no jet-lag protocol.

### 5.10 Sex-specific guidance (Research/13 + `sex.ts`)
- Cycle phase guidance (5 phases × physiology × training note), RED-S screening, hormonal contraception notes, iron deficiency.
- Profile page has no sex / cycle / contraceptive tracking.

### 5.11 Age-specific defaults (Research/14 + `age.ts`)
- `AGE_DEFAULTS_BY_DECADE` — 9 decades × volume / recovery / strength / injury risk.
- Profile has no age. A 60-year-old runner getting "2 quality days a week" guidance is being given the same prescription as a 25-year-old.

### 5.12 Wearable load metrics (Research/15 + `wearables.ts`)
- TRIMP, TSS, monotony, strain, EWMA-ACWR all defined.
- The app computes ACWR (rolling-average, not EWMA). It does not compute monotony or strain.
- Health page shows raw HR / cadence / mileage trends. No load-trend chart, no monotony alert ("you've trained the same load 3 weeks in a row — vary it"), no fitness/freshness curves.

### 5.13 Form / cadence (Research/16 + `cadence.ts`)
- `/runs/[id]:231` displays cadence in spm but doesn't compare to a target band.
- No cadence-trend tile on /health.

### 5.14 Detailed taper specifics (Research/08 + `race_week.ts`)
- 19 race-week constants — race-week meal logistics, sleep banking, kit dress rehearsal, race-day timeline, warm-up specifics by distance.
- The race-week brief variant (`coach.ts:558-563`) mentions "lock in sleep, carbs, and your kit" generically without pulling specifics.

### 5.15 Plan templates surface
- `PLAN_TEMPLATES` is wired into the engine's day picker but the **template itself is never shown to the user**. They can see today's prescription and the next 4 days but cannot see "you're on the marathon-intermediate plan, week 8 of 16, the upcoming peak block is X." There is no plan overview page.

## 6. Gap analysis — info we WANT but don't have

Beyond surfacing existing doctrine, these are the structural gaps where *something new* needs to land (state, research, or UI).

### 6.1 New state CoachState fields needed
- **age, sex, weight (kg)** — required for AGE_DEFAULTS_BY_DECADE, PRE_RACE_MEAL g/kg, sex-specific rules, hydration ml/kg/day.
- **HRmax + LTHR** — replaces the hardcoded 152 bpm. From a max-effort race or user input. Then HR zones become user-specific.
- **Injury flag with type + onset date** — drives `injury_return.ts:WALK_RUN_PROTOCOL`, suppresses quality, suggests cross-training.
- **Travel itinerary (race-bound flights with timezone deltas)** — drives `travel.ts:TRAVEL_ARRIVAL` + jet-lag protocol.
- **Long run day preference + days running per week** — currently `app/profile/page.tsx:90-119` has a UI but local-only.
- **Cycle log** (optional, opt-in) — for sex.ts MENSTRUAL_CYCLE_GUIDANCE.
- **Carb tolerance calibration (g/hr ceiling)** — type definition mentions `carbToleranceDelta` in the FuelingInput interface but no consumer writes or reads it.
- **HRV / RHR / sleep** — fields exist on CoachState already, but `flags.healthKitAvailable` is hardcoded false. The HealthKit pipeline never lands.

### 6.2 New research areas
Most likely already covered by existing doctrine, but not yet doctrine-extracted:
- **Day-of-week placement rules** — the engine's `defaultByDow` is hardcoded; the doctrine doesn't have a "Tue is canonical for threshold" structured constant.
- **Goal-pace negotiation** — when a user's goal is unrealistic (VDOT-implied is 30 min slower), what's the right escalation? Soft prompt? "Are you sure?" gate?
- **Ramp-up rates from zero** — for a true beginner picking up Couch-to-5K. PLAN_TEMPLATES has a `beginner` tier but the lower bound isn't articulated.
- **Race-week carb load by goal time** — fueling.ts has 8-10 g/kg/day but doesn't differentiate by anticipated finish time (4-hr vs 5-hr marathon both load? Probably yes but worth a doctrine entry).
- **Time-of-day training optimization** — Research has nothing on circadian impact beyond travel.

### 6.3 New UI surfaces

- **Plan overview page** — "your training plan, weeks 1-16." Would consume `PLAN_TEMPLATES` properly. Currently the user has Today + Next 4 + dashboard pulse but no full plan view.
- **Onboarding profile** — age/sex/HRmax intake. Right now /profile is exclusively shoes + an unwired long-run-day picker.
- **Health metrics** with non-placeholder data — currently 4 of 4 HealthKit cards say "—". Either pipe in HealthKit or remove the placeholders.
- **Workout detail page that's wired** — `/workout/[date]` is entirely fake. A real version would consume the engine's day data + show structure (warm-up + reps + cool-down breakdown), past-attempt comparison, conditions forecast, suggested shoe.
- **Injury log + RTR walk-run wizard** — the WALK_RUN_PROTOCOL has 8 stages; UI would walk a runner through stages by week.
- **Travel planner** for upcoming races — when did you arrive? Auto-calculates jet lag stage and adjusts the brief.
- **Retrospective view** for past races — `briefRaceMorning` adapts by horizon, but `retrospect` is stubbed and the existing `/api/retrospective` route exists but is wired only via direct manual trigger; the race detail page in debrief mode doesn't yet pull a Coach-voiced narrative.
- **Race-pace-strategy editor** — currently the user can edit goal time and trigger a rebuild but cannot say "use even-effort vs even-split vs negative-split." `paceStrategy` input has this field; no UI.
- **Hydration / electrolyte plan** alongside the gel schedule on race detail.

### 6.4 Cross-references between existing pieces

- The two daily-card surfaces (`page.tsx:CoachTodayCard` and `training/page.tsx:DailyBriefing`) duplicate effort. Consolidate or differentiate intentionally.
- Two phase concepts: `lib/strava-stats.ts:trainingPulse().phase` ("BUILDING / TAPER / RACE MONTH / POST-RACE / BUILDING / BASE BLOCK") and `lib/coach-principles.ts:Phase` ("BASE / BUILD / PEAK / TAPER / BASE_MAINTENANCE / POST_RACE / REBUILD"). The dashboard uses the former, the engine the latter. Pick one.
- Long-run-day picker in /profile is silent — make it write to a settings table that CoachState reads.
- `recovery.today` is wired via `composeVoiceLead` (the Stage A reconciliation), but the "you ran X mi when I had today as rest" voice doesn't propagate back to the week-strip on /training (the cell labels "Done" but doesn't update the rationale on neighboring days).

## 7. Recommendations queue (prioritized)

A top-10 list ordered by the rough cost / value ratio. Each item names the size: **S** = a day or two, **M** = a week-ish, **L** = multi-week.

### 1. Wire age + sex + HRmax into CoachState (M)
- **What**: add age (number), sex (M/F/X), hrMax (bpm), lthr (bpm) to a Postgres `runner_profile` table; surface fields in /profile; have `gatherCoachState()` read them.
- **Why**: unlocks AGE_DEFAULTS_BY_DECADE (Research/14), sex.ts cycle/RED-S/iron rules (Research/13), and replaces the hardcoded 152 bpm threshold (`coach-principles.ts:239`) with a per-runner threshold (`0.80 × hrMax`).
- **Dependencies**: schema migration, profile UI extension, gatherCoachState rewrite.

### 2. Surface hydration + sodium + sweat-rate plan on race detail (S–M)
- **What**: an HydrationTile alongside FuelingTile on `app/races/[slug]/page.tsx` that consumes `hydration.ts` doctrine: pre-race volume + sodium, during-race sodium intake, EAH risk for slow runners, sweat-rate-based plan toggle.
- **Why**: 12 fully-formed hydration constants are inert today. Highest-leverage UNWIRED file; race day is the moment this matters.
- **Dependencies**: only `hydration.ts` (already defined) and minor tile work.

### 3. Build a real /workout/[date] page (M)
- **What**: replace the static placeholder with engine-driven data — pull `coachDaily(state).today` for the requested date, render the `voiceLead`, the deterministic structure (warm-up/main/cool-down derived from workout type), past-attempts (filter Strava activities by `workoutType` similarity), conditions forecast (lat/lon = home location).
- **Why**: removes the most prominent placeholder surface in the app. Currently any link from the week-strip lands on a fake page.
- **Dependencies**: a "home location" field for weather + workout type → structure mapping (which probably needs a doctrine workout-structure file pulled from `workouts.ts`).

### 4. Wire `recovery_protocols.ts` POST_RACE_BY_DISTANCE + REVERSE_TAPER_PROTOCOL (M)
- **What**: replace the engine's hand-coded post-race ladder (`coach-engine.ts:228-278`) with doctrine-driven lookups; show a per-day reverse-taper preview on the race detail debrief tile.
- **Why**: the engine and doctrine disagree on day-by-day post-race specifics (engine has rest-end at day 3 for marathon, doctrine has a more granular reverse-taper). Single source of truth.
- **Dependencies**: doctrine import only; UI tile is straightforward.

### 5. Implement coach.fuelingFor + retire `lib/fueling-claude.ts` direct paths (M)
- **What**: wire `Coach.fuelingFor(input)` per its interface (coach.ts:259-260) with a deterministic body that reads `fueling.ts:RACE_CARB_TARGETS_G_PER_HR`, `GLUCOSE_FRUCTOSE_RATIO`, `PRE_RACE_MEAL`. Pull `lib/fueling-claude.ts` behind it as the optional LLM brain (with the existing system prompt as voice).
- **Why**: the Coach is supposed to be the single entry; this completes the architectural promise. Also surfaces 80-100 g/hr planning target (currently the fueling planner uses a 60g default).
- **Dependencies**: minor. `RACE_CARB_TARGETS_G_PER_HR` already covers the math.

### 6. Build a /plan overview page (S–M)
- **What**: a page that shows the active `PLAN_TEMPLATES` selection — 16-week marathon-intermediate, current week N, the full sample peak week, and an inline "week-by-week" calendar projecting pickRun forward across the remaining build window.
- **Why**: the most expensive doctrine landed (Stage 4) is invisible. Users see today + 4 days + a weekly grid but can't see "I'm in week 8 of 16."
- **Dependencies**: Just routes to the existing `selectActiveTemplate()` + `coach-engine.simulateWeek` extended to N weeks.

### 7. Wire `mental.ts:GOAL_SETTING_FRAMEWORKS` into the race entry form (S)
- **What**: split `goalDisplay` into A / B / C goals on the race form; race detail page shows all three; brief references whichever is appropriate ("the B goal looks right; A is a stretch from current fitness").
- **Why**: A/B/C goals are best-practice-mainstream; the brief has no way today to softly recommend a goal revision because there's no structure for "what's the realistic goal."
- **Dependencies**: race storage schema bump, form UI extension.

### 8. Implement `coach.adjustForReality` + injury intake (L)
- **What**: a UI to log "I'm injured" (type + onset), which writes a flag to runner state. Coach.adjustForReality consults `injury_return.ts:INJURY_CATALOG`, suppresses quality, suggests cross-training (XT_CARRYOVER_MATRIX), prescribes WALK_RUN_PROTOCOL stages on return.
- **Why**: closes the largest "no UI for it" gap. A runner who tweaks their Achilles cannot tell the app today.
- **Dependencies**: schema, profile UI, engine integration, doctrine wiring across 4 files.

### 9. Replace placeholder HealthKit cards with a real readout (L) — OR remove them
- **What**: either build the HealthKit ingest pipeline (iOS app writes JSON to iCloud, server ingests, fields populate) OR drop the 4 placeholder cards from `/health`.
- **Why**: showing 4 dashed-line "M2 placeholder" tiles forever erodes trust in the rest of the page. Especially since the wired CoachState fields (`recovery.hrv7dAvgMs` etc.) advertise capability the engine never uses.
- **Dependencies**: large if building, trivial if cutting. A short-term move would be to hide the section until data flows.

### 10. Compute monotony + strain on /health (S)
- **What**: extend `lib/strava-stats.ts` with monotony (mean weekly load / SD daily load) and strain (weekly load × monotony) per `wearables.ts:TRAINING_LOAD_METRICS`; surface as new tiles on /health with thresholds (>2.0 monotony = warn, high strain = warn).
- **Why**: monotony is a published illness/injury predictor. The data (per-day mileage + HR) is already in cache.
- **Dependencies**: just stats math + tile.

### Notes on what was deliberately NOT prioritized

- **Pacing strategy editor + paceStrategy()** — interesting but the current Minetti-derived pacing in lib/pacing.ts produces good output without runner intervention. Editing strategy is power-user territory.
- **Travel / jet lag** — high doctrine quality but rare event. Worth doing eventually but lower priority than recurring needs.
- **Mental training visualizations (PETTLEP)** — would be a nice race-week tile but soft compared to hydration which has direct race-day consequence.
