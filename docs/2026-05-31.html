The actual bug · why the Activity tab was empty

The Activity tab's empty state on build 126 was the same bug class as the LogWeek.isCurrent fix that shipped in build 121, just one level deeper in the decoder tree. Build 121 made the leaf LogWeek + LogRun structs lenient, but LogState itself + the four LogFilterAxes siblings (added 2026-05-28 for the filter chips) were still on auto-synthesized strict Decodable. A single null slug inside axes.shoes[] would propagate all the way up: LogShoeAxis → throws → [LogShoeAxis] → throws → LogFilterAxes → throws → LogState → throws → fetchLog's try? swallows → returns nil → Activity tab reads zero runs. 

// Before · auto-synthesized strict Decodable
struct LogState: Decodable {
  let today: String          // missing → throws
  let totalRuns: Int         // missing → throws
  let weeks: [LogWeek]       // single bad week row → throws all the way up
  let axes: LogFilterAxes?   // present-but-malformed → decodeIfPresent re-throws
  ...
}

// After · custom init with try?-per-field
init(from decoder: Decoder) throws {
  let c = try decoder.container(keyedBy: CodingKeys.self)
  self.today = try c.decodeIfPresent(String.self, forKey: .today) ?? ""
  self.totalRuns = try c.decodeIfPresent(Int.self, forKey: .totalRuns) ?? 0
  self.weeks = (try? c.decode([LogWeek].self, forKey: .weeks)) ?? []
  self.axes = try? c.decode(LogFilterAxes.self, forKey: .axes)  // catches the re-throw
  ...
}

Avatar fallback · single source of truth

Five views had their own copy-pasted avatarInitials computed property — TodayView · ActivityView · TargetsView · TrainView · ProfileView. Each version had the same final fallback: "first letter of city if full_name is nil". David has city = "Los Angeles" → every header rendered "L" as if his name was Larry.

Promoted a single avatarInitials computed onto ProfileIdentity, dropped the city fallback (city is not identity), and collapsed all five view copies to delegate. The page-header avatar now renders clean when no name is on file; the dedicated profile screen keeps a "FA" (Faff) fallback because the big gradient circle on that screen needs something rendered.

LoadState enum · stop swallowing failures

High-priority #2 from the brief. The fetch lifecycle pattern across the 4 main tabs was try? await API.fetchX() — silently swallowing every failure. Users had no way to distinguish "loading" / "loaded · empty" / "failed offline" — they all rendered the same empty UI. 

// new Components/LoadState.swift
enum LoadState: Equatable {
  case idle           // never fetched, no cache
  case loading        // first fetch in flight, no cached data
  case loaded         // succeeded at least once
  case failed(String) // last fetch errored
}

// FailedLoadBanner · red pill, only shown when state == .failed AND no cache

Adopted in TodayView, HealthView, ActivityView, RunDetailView. Each view's primary fetch (PlanWeek / HealthState / LogState / RunDetail) is now an explicit do-try-catch that drives the lifecycle; secondary fetches stay try?-swallowed (they degrade gracefully via existing UI). The banner only appears when the failure-with-no-cache case applies, so a transient blip on a cached tab stays silent and doesn't pop an alert over real data.
