Multi-tenant
audit

Runcino has been built around David — his data shape, his time zone, his Strava webhook, his Apple Watch. The question this audit answers is the one that determines whether the system can be shipped past a runner-of-one: can the engine serve a runner who is NOT David? Thirteen distinct runner shapes were enumerated — from a cold-start signup with no data, to a Strava-only web user with no Apple Health, to a female runner with cycle tracking, to a Pacific-bound elite — and the engine was probed at the seams where his data shape papers over its assumptions. Findings are graded the same way as the previous audit pass: P0 ships nothing until fixed, P1 is a credible field bug, P2 is degraded UX, P3 is polish.

2026-06-05 Branch: main Working app: web-v2/ 13 runner shapes audited 31 findings
TL;DR
The system is not multi-tenant-ready. A web-only Strava runner with no Apple Health will see a confident READY readout built from a 70-point baseline plus a LOAD pillar — the four recovery pillars (sleep, HRV, RHR, HR-recovery) silently default to plausible numbers while the renderer still says "all quiet · keep doing what you're doing." Every "today" calc for that same runner silently runs in UTC because the timezone capture ladder has no web-side path. The same anti-pattern — null-coalesce-to-plausible-default instead of surface-the-gap-as-UI — recurs in six independent coach modules, four readiness consumers, and the seed adapter. The synthetic-runner bench would not catch any of it: 68% of the plan engine has zero bench coverage, the four "bench" files are pure-logic facsimiles that test the test instead of the engine, and no persona models the data shapes that surfaced these P0s. Ship-readiness verdict: not yet — but the fixes are concentrated and the doctrine changes are clear.

03 Ship blockers · all P0 findings

Seven P0 findings, six are shape-specific to STRAVA_ONLY_WEB_USER (the deep-probed shape), one is the cross-shape cold-start escape-hatch pattern. Each one renders a confidently-shaped lie to a real user.

P0 STRAVA_ONLY_WEB_USER Cold-start escape hatch never fires when LOAD is alive
web-v2/lib/coach/readiness-brief.ts:313-341 · lib/coach/readiness.ts:145-184

LOAD pillar fires from runs alone — breakdownIsEmpty returns false because LOAD has some observed value, and the runner gets a fabricated READY · all systems in their normal band headline derived from a 70-point baseline ± LOAD only.

The building history short-circuit at readiness.ts:178 only fires when ACWR is null. A runner with 4+ weeks of Strava activity has loadAcwr ≥ 0.8, so the short-circuit never fires. readiness-brief.ts:653's breakdownIsEmpty requires every input to be 'no data' — LOAD with 'X.XX ACWR' as observedV makes the predicate return false. Score becomes BASELINE ± LOAD = 67..75 → band 'ready'. The runner sees the green ring on Today (TodayView.tsx:182-189) and 'All quiet · keep doing what you're doing' from health-actions.ts:581-586, with zero sleep/HRV/RHR/HR-recovery readings backing the verdict.

Fix Treat breakdownIsEmpty as "every recovery pillar is no-data" (sleep + HRV + RHR + HR-recovery), NOT every input. When all four recovery pillars are no-data force band='no-data' and route into the coldStart envelope path so the brief renders "Connect Apple Health to start your baseline" instead of a score derived from ACWR plus a 70 baseline.
P0 STRAVA_ONLY_WEB_USER Recovery brief fabricates baseline numbers when state is null
web-v2/lib/coach/recovery-brief.ts:278, 291-293, 308-309

Three independent fallbacks in the same file, each picking a plausible default when CoachState carries nulls. The runner sees a confident "projected morning RHR 60 / baseline 60" — even though the system has zero RHR readings at all.

computeRhrDelta line 308: const baselineBpm = state.rhrBaseline ?? state.rhrCurrent ?? 60 — a hardcoded 60 bpm fallback. computeHrvRebound line 291-293: baseline = state.hrvBaseline ?? 0; current = state.hrvCurrent ?? baseline; currentDrop = Math.max(0, 0 - 0) = 0 — claims "HRV at baseline, currentDrop = 0" for a runner with zero HRV samples. computeSleepTarget line 278: personalAvg = state.sleep7Avg ?? hoursTarget — claims runner is on target when there's no sleep data.

Fix Return null from loadRecoveryBrief when state.rhrBaseline AND state.hrvBaseline AND state.sleep7Avg are ALL null — no recovery story is honest without any recovery signal. Or short-circuit each pillar individually with a noData: true flag the renderer must check before showing the projection.
P0 STRAVA_ONLY_WEB_USER Web-only Strava runner silently runs in UTC
web-v2/lib/runtime/runner-tz.ts:51-63 · app/api/auth/strava/callback/route.ts (missing capture)

The runner whose onboarding pre-dated the timezone-required check has profile.timezone=null. Every "today" calc silently runs in UTC, putting RUNs on the wrong calendar day for ~17 hours per day in Pacific time.

runnerToday() line 60 returns 'UTC' when profile.timezone is null. captureTimezoneFromDevice() (line 106-127) only fires from watch + iPhone payloads — there is no Strava callback or web-only path that auto-populates the TZ. Once null, every downstream module pulling runnerToday() (recovery-phase, training-form, readiness-brief, plan adapter, score-trend, decoupling-trend, glance-state) operates in UTC: a Pacific runner's Sunday long run lands on Monday's plan slot from midnight UTC (= 5pm Pacific Sunday) through 7am Pacific Monday.

Fix On Strava OAuth callback, accept and persist a tz query param from the client (e.g. Intl.DateTimeFormat().resolvedOptions().timeZone written by a tiny script on the connect-button page). Backfill profile.timezone NOT NULL constraint, then default Pacific tz for any legacy null rows OR fall back to a ProfileGapCard "Set your timezone" nudge.
P0 CROSS-SHAPE PATTERN Null-coalesce-to-plausible-default · the recurring anti-pattern
readiness.ts:178-184 · recovery-brief.ts:278,291-293,308-309 · heat-acclim.ts:83 · health-state.ts:691-695 · seed.ts:1163-1166,1191-1212

Every recovery module independently picks a hardcoded "normal" value when CoachState carries nulls: 60 bpm RHR, 0ms HRV drop = baseline, 60ms HRV center, 50 bpm RHR center, 180lb weight, 50 VO2, 7.5h sleep target as personal avg, watchMode='steady', rhrTrend='plateauing'. The runner sees a confidently-rendered narrative built from numbers the system never observed.

The doctrine gap: NoData is encoded everywhere as either null, 0, undefined, or "", and each consumer handles it with its own ad-hoc fallback. There is no shared presence(state, pillar) predicate.

Fix Define one canonical helper pillarStatus(state, key): 'observed' | 'no-data' | 'building-history' in lib/coach/state-presence.ts. Each consumer must call it before computing anything. When status='no-data', return null / skip the row — never substitute a plausible default. Add a lint rule that flags ?? 0, || 60, || 50 literal-fallback patterns inside lib/coach/* unless paired with an explicit noData branch.
P0 CROSS-SHAPE PATTERN Cold-start fires only when EVERY pillar is null — wrong definition
readiness-brief.ts:313+653-656 · readiness.ts:178-184 · health-actions.ts:581-586

The cold-start escape hatch only fires when EVERY input is null, but LOAD (and any non-recovery pillar) can fire from Strava alone. The "all pillars missing" definition of cold-start is wrong. Cold-start in the runner-facing sense means "no recovery signal," not "no signal at all." Same wrong-definition logic surfaces in breakdownIsEmpty, buildHealthActions empty-state branch, and the readiness score gating — they all collude to ship 'READY · all quiet' for a runner with zero recovery data.

The doctrine gap is conceptual: there is no separation between SIGNAL pillars (LOAD / run history) vs RECOVERY pillars (sleep, HRV, RHR, HR-recovery). Readiness score should require at least one recovery pillar to be non-null before producing a band.

Fix Split CoachState pillars into signalPillars (LOAD) vs recoveryPillars (sleep, hrv, rhr, hr_recovery). Replace breakdownIsEmpty with hasNoRecoverySignal(b) checking only the 4 recovery pillars. Gate the readiness score: if hasNoRecoverySignal, return band='no-data' and route through coldStart envelope regardless of LOAD weight. The empty-state branch in health-actions.ts must check the same predicate and emit "No recovery signal yet · connect Apple Health" instead of "all quiet."
P0 CROSS-SHAPE PATTERN Timezone capture is device-only · web has no path
runner-tz.ts:51-63 · auth/strava/callback/route.ts (missing) · onboarding/complete/route.ts:165-169

The auto-detect ladder has a hole: watch payload, iPhone HK sync, and onboarding all populate it, but a web-only Strava runner's profile.timezone stays null and runnerToday() silently falls back to UTC. This corrupts every "today" read downstream (readiness, plan adapter, glance-state, recovery-phase, training-form, score-trend, decoupling-trend, day-of-week patterns, ALL crons).

The doctrine gap: there's no client-side capture on the web (a one-line Intl.DateTimeFormat().resolvedOptions().timeZone would work) and no nightly backfill cron checking for null timezones.

Fix (1) On the /connect/strava page, write a tiny script that POSTs the browser's resolved TZ to /api/profile/timezone before redirecting to OAuth. (2) Add a timezone NOT NULL constraint with a backfill defaulting any remaining nulls to 'America/Los_Angeles'. (3) Surface a ProfileGapCard "Confirm your timezone" nudge on Today if profile.timezone matches the default-backfill marker. (4) Add an assertion in runnerToday() that logs a warning when tz === 'UTC' for diagnostic visibility.
P0 CROSS-SHAPE PATTERN Zero is overloaded as both a real value AND absence
seed.ts:1163, 1180-1212 · TodayView.tsx:543-558

The seed adapter coerces null → 0 for sleep / HRV / RHR / weight / VO2 / wristTemp / spo2 before downstream consumers. TodayView (TodayView.tsx:543 sleepTile.current ?? null resolves 0, not null!) and the body-tile chart domain logic both treat 0 as a valid measurement — contradicting the noData flag the same row carries. Two truths in the same payload.

Concrete: seed.ts:1163 lastNightSleep = ... ?? 0. seed.ts:1180-1181 bfCurrent = health?.bodyFat.current ?? 0; lmCurrent = health?.leanMass.current ?? 0. mk() emits noData=!hasSleep but current=0. TodayView.tsx:553-558 avg7 != null is TRUE for 0, so the "0.0h · 7-night avg" subtitle prints next to a formatSleep em-dash. Chart domain at seed.ts:1192 takes (hrvCurrent || 60) - 30 → 60ms centerpoint when hrvCurrent is 0/null.

Fix Change all ?? 0 and || <default> patterns in seed.ts adaptHealth to ?? null and ?? undefined. mk() must accept current: number | null and skip rendering numeric chrome when null. Chart domain helper: chartDomain(value, span) = value == null ? [0, 1] : [value - span, value + span]. Audit every current ?? 0 site in the seed (wristTempCurrent, hrvCurrent, rhrCurrent, weightCurrent, vo2Current) and replace — null carries the noData truth.

05 Synthetic suite gaps

The synthetic-runner bench at web-v2/lib/plan/synthetic-runners.ts was built to validate the plan engine against a population of personas. The audit asked: does the bench cover the shapes the engine has to serve, and does it actually exercise the engine? The answer to both is no.

P1 · 8 personas, but all male, all en-US TZ, all the same training week
web-v2/lib/plan/synthetic-runners.ts:70-349
Issue

Diversity is in distance/level/health-pattern only, not in the population dimensions that actually drive the engine's per-runner branches. Zero distinct sex. Zero non-US TZ. Zero menstrual cycle. Zero crossModes. Zero alternative schedule. No 10K at any tier. No elite tier. No persona with measured HR vs persona with null HR. No master-runner (50+) and no youth (sub-20).

The 8 personas cover: beginner-5k, david-sub-1-30-hm, david-mid-block, intermediate-hm-sub-2, advanced-marathon, advanced-plus-ultra, returning-from-injury, sleep-debt-prone. Everyone runs Sun-long / Sat-rest / Tue+Thu-quality.

Fix
Add the 6-8 personas listed in the next finding so the suite spans the actual user-population the engine has to serve, not just a "happy-path matrix of distance × ability."
P1 · healthPattern is dead metadata
web-v2/lib/plan/synthetic-runners.ts:67
Issue

Declared on every persona but never read by any test or bench. "sleep-debt-prone" and "returning-from-injury" are not actually exercised against a stress event-stream — they behave identically to baseline in every test. Grep across the entire web-v2 tree finds healthPattern only at the type def and the literal assignments. Zero reads. Same applies to ageYears: bench treats age 32 and age 42 identically.

Fix
Either delete the dead fields, or build a synthetic event-stream generator (e.g. ./bench-fixtures/health-streams.ts) that emits 14-28d of HRV/RHR/sleep/RPE for each healthPattern, then feed it into detectAdaptations() and detectDrift() and assert the right trigger fires. Same for ageYears: actually pipe it through HRmax derivation paths.
P1 · Plan-engine bench uses an ideal-trajectory shortcut
web-v2/lib/plan/plan-engine.test.ts:32-71
Issue

The bench does NOT run personas through "generate plan → adapt → simulate". buildPersonaInput() hand-rolls a SimulatorInput from expectedPlan.peakWeeklyMileageBand (averaging the band's lo+hi), feeding the simulator an ideal trajectory the generator never produced. composePlan() is never called from plan-engine.test.ts. The bench validates ONLY simulator math against a hand-built ramp; it cannot catch generator bugs that produce non-ideal ramps.

No bench file imports detectAdaptations, applyAdaptations, detectDrift, adaptToday, simulateActivePlan (the DB version), composeMaintenancePlan, composeRecoveryPlan, generatePlan (the seeding wrapper), or autoRebuild.

Fix
Add a true end-to-end bench (e2e-bench.test.ts) that: 1) calls composePlan(personaToInput(p)), 2) injects synthetic completed runs from a healthPattern event stream, 3) calls detectDrift + detectAdaptations against an in-memory snapshot, 4) re-runs simulate() on the post-adaptation plan, 5) asserts goal-gap.classifyGoalGap() returns the expected status.
P1 · Adapter / adaptive-ramp benches test pure-logic facsimiles, not the actual code
adapter-bench.test.ts:23-182 · adaptive-ramp.test.ts:23-192
Issue

The benches define inline copies of the adapter's math (simShave, allGatesPass, applyBump) and test those copies. The actual adapt.ts and adaptive-ramp.ts code paths are never called. A regression in the real SQL/UPDATE path will not be caught.

06 Cross-shape patterns · the higher-order bugs

PATTERN 1 · P0 · Null-coalesce-to-plausible-default
readiness.ts · recovery-brief.ts · heat-acclim.ts · health-state.ts · seed.ts (6+ files)
The pattern

Fix
One canonical pillarStatus(state, key) helper in lib/coach/state-presence.ts, called by every consumer. Lint rule flags ?? 0, || 60, || 50 literal-fallback patterns inside lib/coach/*.
PATTERN 2 · P0 · Cold-start = "no signal at all" is the wrong definition
readiness-brief.ts:313+653-656 · readiness.ts:178-184 · health-actions.ts:581-586
The pattern
Cold-start should mean "no recovery signal," not "no signal at all." There is no separation between SIGNAL pillars (LOAD / run history) vs RECOVERY pillars (sleep, HRV, RHR, HR-recovery). Readiness score should require at least one recovery pillar to be non-null before producing a band.
Fix
Split CoachState pillars. Replace breakdownIsEmpty with hasNoRecoverySignal checking only the 4 recovery pillars. Gate the score: if no recovery signal, force band='no-data' regardless of LOAD weight.

PATTERN 6 · P1 · Readiness score has no coverage field
readiness.ts:38 (BASELINE=70) · weighted-pillar fan-in

PATTERN 7 · P1 · Fallback ladders end in value-shaped sentinels instead of UI nudges
runner-tz.ts (UTC) · health-state.ts (steady) · heat-acclim.ts (plateauing) · etc.
The pattern
Auto-detection assumes at least one rich device is connected. Strava-only web runner is an unmodeled persona — no fallback ladder accounts for "web browser is the only entry point." Same recurs for max HR, VO2, timezone. Ladders top out at "we have no device" and either silently default or render NULL. Neither is the right UX outcome. No shared "PromptForInput" contract.
Fix
Define resolveOrPrompt<T>(ladder: Source[], onExhausted: GapPrompt): T | GapPrompt in lib/coach/resolve.ts. Each resolver returns either a value or a typed gap-prompt the renderer turns into a ProfileGapCard ("Set your timezone" / "Connect Apple Health for HRV" / "Manually enter resting HR"). Plumb up to TodayView's gap-card slot. Stop letting modules silently degrade.
PATTERN 8 · P2 · Each module defines "data exists" independently

07 Next 5 moves to ship

If the goal is "system can serve a runner who is NOT David" — and ship in that order — here is the recommended sequence. Each move is concentrated, has a small blast radius, and unlocks multiple downstream findings at once.

1
Build the presence module
Create web-v2/lib/coach/state-presence.ts with four canonical predicates: hasRecoverySignal(state), hasSleepSignal(state), hasHrvSignal(state), hasRhrSignal(state). Refactor every consumer (readiness scorer, recovery brief, heat acclim, watchMode, health-actions empty-state, seed adapter) to call them. This is the load-bearing fix — it kills 3 of the 7 P0s (Patterns 1, 2, and the cold-start escape hatch) and 5 of the 12 P1s at the same time. Add a lint rule banning ?? 0, || 60, || 50 literal-fallbacks inside lib/coach/*.
2
Close the web-side timezone gap
On the /connect/strava page (or wherever the Strava OAuth button lives), capture Intl.DateTimeFormat().resolvedOptions().timeZone client-side and POST to /api/profile/timezone before redirect. Add a timezone NOT NULL migration with a backfill default (David's TZ). Add a nightly cron checking for unset/default-only timezones and surfacing a ProfileGapCard. Add a diagnostic warning when runnerToday() resolves to 'UTC'. Closes Pattern 3 plus eliminates the silent UTC drift affecting every non-David shape.
3
Kill the seed adapter's null-to-zero coercion
In web-v2/components/faff-app/seed.ts adaptHealth, change every ?? 0 and || <default> to ?? null. Update mk() to accept current: number | null and skip rendering numeric chrome when null. Add a chartDomain(value, span) helper that returns [0, 1] when value is null. Closes Pattern 4 and unblocks Today + Health rendering for any partial-data shape.
4
Add the 4 data-shape personas to the synthetic suite
Add strava-only-web, watch-only-iphone, manual-only, and mixed-partial personas to web-v2/lib/plan/synthetic-runners.ts. Build an event-stream fixture generator that emits 14-28d of HRV/RHR/sleep/RPE per healthPattern. Wire them into a new e2e-bench.test.ts that calls composePlan → detectDrift + detectAdaptations → simulate end-to-end. CI must fail if any of these renders 'READY' or 'all quiet' on a cold-start envelope. Stops the regression from re-emerging the next time a recovery pillar gets added.
5
Enforce "cite or shut up" on authored prose
Annotate every authored message template in recovery-brief.ts, health-actions.ts, heat-acclimatization.ts, and the readiness-brief composer with a requires: PillarKey[] contract. Render layer compares against pillarStatus and falls back to a noData stub when any required input is absent. CI test walks every template and asserts requires is non-empty (or explicitly marked [] with a justification comment). Closes Pattern 5 plus eliminates fabricated prose from heat-acclim and recovery-brief permanently.

These five moves take the system from "works for David" to "works for any runner who shares David's instrumentation richness OR connects only Strava OR runs in a non-US TZ." They do not solve the higher-order shape coverage (active injury, female cycle tracking, treadmill-only, post-A-race recovery, multi-race horizon) — those are downstream work that gets easier once the presence module, TZ capture, and seed-null patterns are fixed. The five moves are roughly 1-2 days of work each. None of them require new doctrine — just disciplined application of doctrine that already exists.
