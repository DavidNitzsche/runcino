09 What surprised me

  1. The biggest gap is not code — it's operational. The detectors, the orchestrator, the action-applier, the writeIntent helper, the bustCache hook — all exist and look correct. The cron file exists. The GH Action file exists. But training_plans.last_adapted_at is NULL on every plan and coach_actions has 2 rows total, both from manual seeds in May. Something between GH Actions and the production endpoint is broken and has been broken for the entire post-cutover period.
  2. Two crons have no scheduler at all. notifications and snapshot-projections have endpoint files, route docstrings claiming "runs every 15 min via cron-job.org," and zero evidence anyone POSTs to them. The previous audit graded these features as "shipped." They're cold.
  3. buildInjuryPlan is dead. The injury builder was a deliberate piece of work — 200+ lines, walk-run scaffolding, cross-training rotation, taper logic. It compiles, type-checks, exports cleanly. Nothing calls it. The injury trigger fires an entirely different code path (a downgrade with a copy-pasted message). This is the worst kind of "shipped" — the file is there for everyone to point at, the runtime behavior is unchanged.
  4. The watch endpoints are wide open. The watch is supposedly proxied through iOS, so the watch routes can accept ?user_id= from the request itself. With session auth in front of every other endpoint, this is an obvious leak vector for user #2. The previous audit's per-client matrix glossed over this.
  5. Weather enrichment is much better than CRON_AUDIT claimed. 71/126 enriched vs the doc's 5% snapshot — looks like the cron has been catching up since. So at least one of the "broken" things in the previous audit has self-healed.
  6. Cold-start isolation actually works. Despite all the back-compat hazards above, the probe-verified cold-start path is genuinely clean. 0/22 leak surfaces. The pieces that were claimed to be fixed at the L1 isolation level are fixed. The remaining isolation risk is one rung up — the route-level default-to-David fallback when no auth is present.

10 Recommended P0 priority order

  1. Fix run-adaptations cron in production. Check the GH Actions tab for the last 14 days of run-adaptations workflow runs. If they're red, find out why. If they're green but the route is rejecting, dump the request log. Without this, the "coach is mastermind" doctrine is a slogan.
  2. Add GH workflow for notifications. Pattern after refresh-briefings.yml. Schedule every 15 min. Then drain the 2 stuck rows manually.
  3. Add GH workflow for snapshot-projections. Daily 07:30 UTC. Then re-validate the RaceHeader projectionTrend with real cron-sourced rows.
  4. Wire buildInjuryPlan into the injury trigger. When detectInjuryActive fires, the action should call buildInjuryPlan and replace plan_workouts for the affected window, not just emit a downgrade string.
  5. Add a session check to the watch routes. Even if the watch uses a long-lived service token, the route should validate the token and reject if it doesn't match the body's user_id.
